## 1. Abstract

This RFC defines the implementation-level distinction between **Tool Calls** and **Subagent Calls** in NoeAgent. While [RFC-1003](RFC-1003.md) establishes the abstract capability type taxonomy (TOOL, MCP_TOOL, SKILL, AGENT, CLI_AGENT), this RFC specifies how NoeAgent decides between tool and subagent execution paths, and how each modality is implemented within the LangGraph workflow.

The distinction is foundational: a **Tool** extends NoeAgent's capability; a **Subagent** extends NoeAgent's cognition.

## 3. Background & Motivation

NoeAgent can fulfill plan steps using either **tool calls** or **subagent calls**. Without clear architectural boundaries, implementations risk:

* Hidden state in tools (violating referential transparency)
* Subagent overuse for trivial operations (resource waste)
* Tool underuse for complex tasks (lost delegation opportunities)
* Confused failure recovery strategies

This RFC provides the decision framework and implementation contract to ensure clean separation between the two execution modalities.

---

## 4. Design Principles

1. **Ontological Clarity**: A tool is an extension of capability; a subagent is an extension of cognition.
2. **Referential Transparency**: Tools MUST be stateless and referentially transparent when possible; subagents are intentionally not.
3. **Delegation of Reasoning**: Tool calls retain caller control; subagent calls delegate autonomous reasoning.
4. **Cost Proportionality**: Use tools by default; escalate to subagents only when complexity warrants the cost.
5. **Explicit Boundaries**: The decision between tool and subagent MUST be traceable in the execution graph.

---

## 5. Ontological Distinction

### 5.1 Conceptual Model

| Dimension | Tool Call | Subagent Call |
|-----------|----------|---------------|
| Nature | Function / procedure | Cognitive worker |
| Identity | Stateless capability | Stateful actor |
| Abstraction Level | Procedure | Reasoning entity |
| Role | Execute | Think + Execute |

### 5.2 Design Invariant

> **A tool extends NoeAgent's capability. A subagent extends NoeAgent's cognition.**

Tools are passive executors. Subagents are active reasoners.

---

## 6. Control Model

### 6.1 Tool Control Flow

Caller retains full control. The tool does not decide what to do next.

```
execute_step → AgentAction.tool_calls → tool_node → ToolMessage → execute_step
```

Characteristics:
* **Deterministic**: Given same input → same output (ideally)
* **Single-turn**: Request-response atomic
* **Caller-owned reasoning**: NoeAgent owns all decision-making
* **Passive execution**: Tool awaits invocation

### 6.2 Subagent Control Flow

Delegated autonomy. The subagent is allowed to think inside its boundary.

```
execute_step → AgentAction.subagent → subagent_node
                                        ↳ child.arun() → internal loop
                                        ↳ child tool usage
                                        ↳ child planning
                                      → result → execute_step
```

Characteristics:
* **Multi-step internal loop**: Subagent may iterate
* **Internal planning**: Subagent may decompose tasks
* **Tool access**: Subagent may invoke tools
* **Owns partial reasoning space**: Subagent has autonomous cognition

---

## 7. State and Memory Model

### 7.1 Comparison Table

| Aspect | Tool | Subagent |
|--------|------|----------|
| Stateful | No | Yes |
| Session Memory | None | Persistent (within session) |
| Cross-call continuity | No | Yes |
| Internal scratchpad | No | Yes |

### 7.2 Design Invariant

**Tools MUST be referentially transparent when possible. Subagents are intentionally not.**

External CLI daemons (per [RFC-1003](RFC-1003.md) §10) amplify this distinction—they maintain process-level state across multiple task dispatches.

---

## 8. Temporal Scope

### 8.1 Tool Temporal Scope

* **Instantaneous**: Bounded execution time
* **Blocking**: From NoeAgent's perspective
* **Request-response atomic**: Single round-trip

### 8.2 Subagent Temporal Scope

* **Long-lived**: May run for seconds to minutes
* **May be asynchronous**: Non-blocking from caller perspective
* **Lifecycle states**: BUSY / IDLE transitions
* **Owns lifecycle**: Independent lifecycle management

### 8.3 System Impact

This difference impacts:
* Scheduler design
* Resource accounting
* Failure recovery
* Timeout handling

---

## 9. Failure Semantics

### 9.1 Comparison Table

| Aspect | Tool | Subagent |
|--------|------|----------|
| Crash impact | Single request | Session-level |
| Recovery | Retry call | Restart process |
| State loss | None (stateless) | Possible |
| Health monitoring | Optional | Mandatory (for CLI daemons) |

### 9.2 Recovery Requirements

* **Tools**: Simple retry with backoff
* **Subagents**: Require heartbeat, liveness detection, and restart strategy per [RFC-1003](RFC-1003.md) §11

---

## 10. Scheduling Philosophy

### 10.1 Tool = Cheap Primitive

* Spawn per call, no lifecycle management
* No queuing needed
* Parallelizable (multiple `ToolCallAction` in one `AgentAction`)
* Resource cost: negligible

### 10.2 Subagent = Heavy Worker

* Reuse instance across tasks
* Task queue per instance
* Concurrency policy required
* Resource cost: significant (LLM tokens, process memory)

---

## 11. NoeAgent Decision Matrix

### 11.1 Routing Heuristics

When `TaskStep.execution_hint` is `"auto"`, the planner and LLM use these heuristics:

| Signal | Route to | Example |
|--------|----------|---------|
| Single atomic operation | **Tool** | `web_search("python asyncio")` |
| Read/write a file | **Tool** | `file_edit.read_file("config.py")` |
| Run a shell command | **Tool** | `bash.run_bash("ls -la")` |
| Multi-step research with synthesis | **Subagent** | "Analyze the auth module" |
| Independent parallel subtasks | **Multiple subagents** | "Search 3 databases concurrently" |
| Task requiring persistent code context | **CLI subagent** | "Refactor the payment module" |
| Specialized external agent capability | **CLI subagent** | "Claude: review for security issues" |
| Simple computation | **Tool** | `python_executor("sum(range(100))")` |
| Complex analysis needing planning | **Subagent** | "Compare 5 frameworks and recommend" |
| Latency-sensitive, low complexity | **Tool** | Quick search/lookup |
| Quality-sensitive, high complexity | **Subagent** | Deep research/analysis |

### 11.2 Default Behavior

**Use tools by default. Escalate to subagents only when the task requires autonomous multi-step reasoning, persistent state, or specialized external capabilities.**

## 12. Implementation Mapping

### 12.1 AgentAction Schema

```python
class AgentAction(BaseModel):
    thought: str
    tool_calls: list[ToolCallAction] | None = None
    subagent: SubagentAction | None = None
    text_response: str | None = None
    mark_step_complete: bool = False
```

**Constraint**: Exactly one of `tool_calls`, `subagent`, or `text_response` MUST be set (enforced by `model_validator`).

### 12.2 Routing in _route_after_execute

```python
def _route_after_execute(self, state: AgentState) -> str:
    if plan.is_complete: return "finalize"
    if last_msg has tool_calls: return "tool_node"
    if last_msg has subagent_action: return "subagent_node"
    if iteration >= max: return "finalize"
    if reflection_due: return "reflect"
    return "execute_step"
```

### 12.3 tool_node Implementation

```
For each ToolCallAction:
  1. Pre-validate args against tool.input_schema
  2. Apply type coercion if needed
  3. Dispatch via ToolExecutor.run()
  4. Collect result as ToolMessage
  5. Enforce max_tool_calls_per_step limit
```

### 12.4 subagent_node Implementation

```
Read subagent_action from last message:
  spawn → NoeAgent child (in-process) or ExternalCliAdapter.spawn()
  interact → child.arun() or adapter.send()
  spawn_cli → ExternalCliAdapter.spawn(CliSubagentConfig)
  interact_cli → adapter.send(handle, message)
  terminate_cli → adapter.terminate(handle)
```

### 12.5 Subagent-as-Tool Fallback

When `enable_subagents=True`, a `spawn_subagent(name, task, mode)` function is registered as a regular tool. This allows the LLM to use the standard tool-calling path even for subagent tasks. `tool_node` also intercepts `tool_name="subagent"` and redirects to `spawn_subagent`.

## 13. Anti-Patterns

These patterns indicate architectural confusion between tools and subagents:

| Anti-Pattern | Problem | Fix |
|-------------|---------|-----|
| Hidden state in tools | Tool maintains internal state across calls | Make tool stateless; use subagent if state needed |
| Unbounded tool runtime | Tool runs for minutes without timeout | Set timeout; consider subagent for long tasks |
| Subagent for trivial ops | Spawning a subagent to read a file | Use tool directly |
| Tool that plans internally | Tool does multi-step reasoning | This is a subagent; model it correctly |
| Single-use subagent for atomic task | Spawn + interact + terminate for one search | Use tool instead |

## 14. Cost and Performance Considerations

### 14.1 Metrics Comparison

| Metric | Tool | Subagent |
|--------|------|----------|
| Latency (typical) | 100ms - 10s | 10s - 300s |
| LLM tokens per invocation | 0 (tool itself) | 1000-50000+ (child LLM calls) |
| Memory overhead | None | Child agent state + memory providers |
| Process overhead | None | Child NoeAgent or CLI daemon process |
| Parallelization | Multiple `ToolCallAction` in one step | Multiple `spawn_subagent` calls |

### 14.2 Resource Guidelines

* **Tool**: Cheap, parallelizable, stateless
* **Subagent**: Expensive, sequential (per instance), stateful

**Guideline**: Use tools by default. Escalate to subagents only when the task requires autonomous multi-step reasoning, persistent state, or specialized external capabilities.

## 15. Observability

### 15.1 Tool Observability

* `TOOL_START` / `TOOL_END` progress events
* Input args and output result logged to session JSONL

### 15.2 Subagent Observability

* `SUBAGENT_START` / `SUBAGENT_PROGRESS` / `SUBAGENT_END` progress events
* Child agent has its own session log (nested JSONL)
* Internal reasoning trace available in child's session log
* CLI daemon stdout captured by `ExternalCliAdapter`

## 17. Examples

### 17.1 Tool Call Example

```python
# User request: "Search for recent papers on transformers"
# NoeAgent decision: Single atomic operation → Tool

AgentAction(
    thought="Need to search for transformer papers",
    tool_calls=[
        ToolCallAction(
            tool_name="web_search",
            args={"query": "transformer papers 2024"}
        )
    ]
)
```

### 17.2 Subagent Call Example

```python
# User request: "Analyze the authentication module for security issues"
# NoeAgent decision: Multi-step analysis with synthesis → Subagent

AgentAction(
    thought="Security analysis requires multi-step reasoning",
    subagent=SubagentAction(
        action="spawn",
        name="security-analyzer",
        task="Analyze authentication module for security vulnerabilities"
    )
)
```

### 17.3 Parallel Tool Calls Example

```python
# User request: "Get the weather in NYC, London, and Tokyo"
# NoeAgent decision: Multiple independent atomic operations → Parallel tools

AgentAction(
    thought="Three independent lookups can run in parallel",
    tool_calls=[
        ToolCallAction(tool_name="weather", args={"city": "NYC"}),
        ToolCallAction(tool_name="weather", args={"city": "London"}),
        ToolCallAction(tool_name="weather", args={"city": "Tokyo"}),
    ]
)
```

## 19. Conclusion

RFC-1005 establishes the implementation-level distinction between tool calls and subagent calls in NoeAgent. The key insights are:

* **Ontological distinction**: Tools extend capability; subagents extend cognition.
* **Control flow**: Tools are passive executors; subagents are autonomous reasoners.
* **Cost proportionality**: Use tools by default; escalate to subagents for complexity.
* **Explicit routing**: Decision is traceable in the execution graph.

> **Every tool call is a procedure invocation. Every subagent call is a reasoning delegation.**
