# RFC-9004: Universal Subagent Interface and Dynamic Registry

## 1. Abstract

This RFC proposes a unified, event-driven `Subagent` interface and a dynamic registry for the Noesium core framework. Currently, in-process and CLI subagents are hardcoded into the orchestrator (`NoeAgent`) via ad-hoc factory methods, and lack standardized mechanisms for lifecycle management, event streaming, context sharing, and Human-In-The-Loop (HITL) interactions. By defining a robust protocol that all subagents must implement and a plugin architecture for discovery, this design enables scalable integration of future subagents.

---

## 2. Scope and Non-Goals

### 2.1 Scope

This RFC defines:
* A universal `SubagentProtocol` in Python handling lifecycle, invocation, and context.
* Event streaming interfaces allowing subagents to publish intermediate thoughts, tool usage, and progress.
* Standardized HITL pause/resume mechanisms.
* A dynamic plugin/registry system to dynamically load subagents, removing hardcoded references from `NoeAgent._setup_builtin_subagents`.
* An adapter pattern integrating both in-process and out-of-process (CLI/RPC) subagents into this unified model.

### 2.2 Non-Goals

This RFC does **not** define:
* Implementation details of specific new subagents (e.g., changes to internal BrowserUseAgent logic).
* The underlying network transport for remote subagents (beyond defining how they adapt to the protocol).
* Complete rewrite of the existing Capability Registry (RFC-1004), but rather how subagents interface with it.

---

## 3. Background & Motivation

In the current implementation, `NoeAgent` manages built-in subagents (`BrowserUseAgent`, `TacitusAgent`) through hardcoded factory methods (`_create_browser_use_agent`, `_create_tacitus_agent`) and a brittle `BuiltInAgentCapabilityProvider` that attempts to guess the execution method (`arun`, `run`, or `research`).

Furthermore:
1. **Opaque Execution**: Subagents execute as black boxes, making it difficult to stream intermediate thoughts or tool usage back to the parent agent or UI.
2. **Context Isolation**: Sharing memory or context between the parent orchestrator and the subagent requires ad-hoc data passing.
3. **No HITL Standardization**: If a subagent needs human input (e.g., handling CAPTCHAs, confirming dangerous actions), there is no framework-level way to pause the subagent, request input via the parent, and resume.
4. **Poor Extensibility**: Adding a new built-in subagent currently requires modifying core `NoeAgent` code.

A unified, event-driven interface solves these issues by treating all subagents as standardized, stateful event streams.

---

## 4. Design Principles

1. **Protocol-Oriented**: Subagents conform to a strict interface (`SubagentProtocol`), ensuring the orchestrator does not need to know the underlying implementation details.
2. **Event-Driven Execution**: Invocation returns an asynchronous event stream rather than a single blocking result, enabling real-time UI updates and observability.
3. **Context-Aware**: Subagents receive a unified context object containing shared memory, configuration, and parent references.
4. **Plugin-Based Discovery**: Subagents are discovered dynamically via entry points or a configuration-driven registry, eliminating hardcoded orchestrator dependencies.

---

## 5. Proposed Design

### 5.1 The `SubagentProtocol`

A new abstract base class or protocol will be introduced in `noesium.core.agent.subagent`.

```python
from typing import AsyncGenerator, Protocol, Any, Dict
from noesium.core.event.envelope import EventEnvelope

class SubagentContext:
    """Context passed to subagents during initialization and execution."""
    session_id: str
    parent_id: str
    shared_memory: Any
    config: Dict[str, Any]

class SubagentProtocol(Protocol):
    """The universal interface for all subagents."""

    @classmethod
    def get_descriptor(cls) -> "CapabilityDescriptor":
        """Returns the capability descriptor for registration."""
        ...

    async def initialize(self, context: SubagentContext) -> None:
        """Setup lifecycle hook before first execution."""
        ...

    async def shutdown(self) -> None:
        """Cleanup lifecycle hook."""
        ...

    async def execute(self, task: str, **kwargs: Any) -> AsyncGenerator[EventEnvelope, None]:
        """
        Execute the task and yield a stream of events.
        Events can represent thoughts, tool usage, requests for human input, and the final result.
        """
        ...

    async def resume(self, input_data: Any) -> AsyncGenerator[EventEnvelope, None]:
        """Resume execution after yielding a HITL pause event."""
        ...
```

### 5.2 Event Streaming & HITL

Instead of returning a final string, the `execute` method yields `EventEnvelope` objects. The parent orchestrator consumes this stream.

**Key Event Types:**
* `subagent.thought`: Intermediate reasoning.
* `subagent.tool_call`: Emitted when the subagent invokes a tool.
* `subagent.progress`: Percentage or milestone updates.
* `subagent.hitl_request`: Emitted when the subagent pauses execution awaiting human input.
* `subagent.completed`: The final result.
* `subagent.failed`: Execution failure.

**HITL Flow:**
1. Subagent yields `subagent.hitl_request(prompt="Enter CAPTCHA")`.
2. Parent orchestrator catches the event, suspends subagent task processing, and forwards the prompt to the user.
3. User provides input.
4. Orchestrator calls `subagent.resume(input_data="1234")`, returning a new event stream.

### 5.3 Dynamic Registry & Discovery

Remove `_setup_builtin_subagents` from `NoeAgent`. Introduce a `SubagentLoader` that uses Python Entry Points (`importlib.metadata`) or a configuration-driven module loader.

```python
# In noesium/core/capability/loader.py
def discover_subagents() -> list[Type[SubagentProtocol]]:
    # Load via python entry points 'noesium.subagents'
    # Fallback to scanning config.subagent_modules
    ...

# In NoeAgent
async def _setup_subagents(self):
    for subagent_class in discover_subagents():
        provider = UnifiedAgentCapabilityProvider(subagent_class)
        self._registry.register(provider)
```

### 5.4 Adapters for Legacy and Out-of-Process Agents

* **In-Process Adapter**: A wrapper for existing agents (like `BrowserUseAgent` or `TacitusAgent`) that haven't been migrated to `SubagentProtocol`. It translates their `arun`/`research` blocking calls into a single `subagent.completed` event.
* **CLI/External Adapter**: Replaces the current `CliAgentCapabilityProvider`. It parses the NDJSON stdout from the external process and yields standard `EventEnvelope` objects, mapping external lifecycle to `initialize` and `shutdown`.

## 6. Examples

### Migrating `TacitusAgent`

```python
class UnifiedTacitusAgent(SubagentProtocol):
    async def initialize(self, context: SubagentContext):
        self.context = context
        self.llm = get_llm_client()

    async def execute(self, task: str, **kwargs) -> AsyncGenerator[EventEnvelope, None]:
        yield create_event("subagent.progress", data={"status": "Starting research"})

        # Simulated intermediate steps
        yield create_event("subagent.thought", data={"text": "Searching knowledge base"})

        result = await self._perform_research(task)
        yield create_event("subagent.completed", data={"content": result.content})
```

## 7. Alternatives Considered

### 7.1 Standard RPC/REST API Model
* **Description**: Subagents run as distinct microservices exposing REST or gRPC endpoints.
* **Pros**: Language agnostic, strict isolation, highly scalable across machines.
* **Cons**: Massive overhead for lightweight in-process subagents. Managing network failures, auth, and deployments complicates the local developer experience. The event-driven protocol adapter approach allows out-of-process execution without forcing all agents into a microservice model.

### 7.2 Actor Model (e.g., Ray / Pykka)
* **Description**: Every subagent is an Actor passing asynchronous messages to the orchestrator.
* **Pros**: Excellent for massive concurrency and fault tolerance. True isolation of state.
* **Cons**: Introduces heavy third-party dependencies. Re-architecting the entire `NoeAgent` to pure actor message passing is a massive scope creep. The proposed Event Stream (`AsyncGenerator`) pattern natively fits `NoeAgent`'s existing async loop without requiring a full Actor runtime.

## 8. Relationship to Other RFCs

* **RFC-0005**: Expands on the capability model by defining exactly how the `CapabilityType.AGENT` executes.
* **RFC-1004**: Provides the registry where these dynamically loaded `UnifiedAgentCapabilityProvider`s will be stored.

---

## 9. Conclusion

By migrating from hardcoded, blocking factory methods to an event-driven `SubagentProtocol` with dynamic discovery, the framework becomes significantly more extensible. Subagents can now transparently stream thoughts, share context, and request human intervention, paving the way for complex multi-agent workflows.

> **"Subagents are not just blocking functions; they are stateful, observable streams of execution."**
