## 1. Abstract

This RFC proposes a unified, extensible subagent functionality interface for the `noesium/core` framework, derived from deep analysis of current `NoeAgent` implementation patterns. The design standardizes how orchestrator agents discover, select, invoke, stream, monitor, and lifecycle-manage heterogeneous subagents (in-process agents, built-in specialized agents, and external CLI/daemon agents).

The RFC introduces a layered contract:

1. **Subagent Descriptor Contract** for discovery and planning.
2. **Subagent Runtime Contract** for invocation semantics.
3. **Subagent Streaming Contract** for progress/event multiplexing.
4. **Subagent Lifecycle Contract** for initialization, health, and teardown.
5. **Subagent Policy Contract** for routing constraints, permissions, and depth limits.

It also evaluates alternatives and recommends a provider-backed interface integrated with `CapabilityRegistry` for maximal compatibility and future growth.

---

## 2. Context: Observed NoeAgent Subagent Patterns

Current `NoeAgent` already supports multiple subagent modalities:

- **In-process child NoeAgent** delegation (`spawn_subagent`, `interact_with_subagent`)
- **Built-in specialized agents** (`BuiltInAgentCapabilityProvider`, e.g., browser_use, tacitus)
- **External CLI subagents** via adapter (`CliAgentCapabilityProvider`)
- **LLM-routed delegation** via `SubagentAction` in structured output
- **Registry-mediated invocation** via `CapabilityRegistry` providers
- **Progress streaming bridge** via `ProgressEvent` queue multiplexing

Strengths in the current model:

- Reuses existing capability architecture (RFC-1004 aligned)
- Preserves subagent-type flexibility
- Supports real-time subagent progress in parent stream

Current friction points for future scalability:

1. **Action semantics are stringly-typed** (`spawn`, `invoke_builtin`, `invoke_cli`, etc.).
2. **Invocation paths differ by subtype** (conditional logic in `subagent_node`).
3. **Capabilities and runtime contracts are not explicitly unified** across subagent types.
4. **Lifecycle boundaries** (init/cleanup/health/cancellation) are implicit and scattered.
5. **Policy constraints** (`requires_explicit_command`, depth, permissions) are ad hoc.

## 3. Goals and Non-Goals

### 3.1 Goals

- Define a **single core subagent interface** usable by all orchestrator agents.
- Preserve compatibility with current NoeAgent behavior and `CapabilityRegistry`.
- Support **multiple execution backends** (in-proc, CLI, remote service).
- Standardize **streaming, cancellation, and error semantics**.
- Encode **routing policy metadata** for planner and runtime enforcement.
- Enable safe incremental adoption without a full rewrite.

### 3.2 Non-Goals

- Replacing existing tool contracts in RFC-2004.
- Defining UI-level behavior (TUI/Frontend rendering specifics).
- Mandating one planning strategy for all agents.
- Implementing all migration work in this RFC.

---

## 4. Design Overview

Introduce a new core abstraction set under `noesium/core/agent/subagent/`:

- `SubagentDescriptor` (static metadata)
- `SubagentInvocationRequest` / `SubagentInvocationResult` (runtime I/O)
- `SubagentProgressEvent` (stream protocol)
- `BaseSubagentRuntime` (execution interface)
- `SubagentProvider` (registration and lazy factory)
- `SubagentManager` (selection + invocation + lifecycle)

High-level flow:

```
Planner/Router -> SubagentManager.select() -> SubagentProvider
                                         -> runtime.invoke() / runtime.invoke_stream()
                                         -> normalized events/results
                                         -> parent agent stream + memory/logging
```

## 5. Core Interface Specification

### 5.1 SubagentDescriptor

Required fields:

- `subagent_id: str` (stable technical identifier)
- `display_name: str`
- `description: str`
- `backend_type: Literal["inproc", "builtin", "cli", "remote"]`
- `task_types: list[str]`
- `keywords: list[str]`
- `requires_explicit_command: bool`
- `supports_streaming: bool`
- `supports_parallel_invocation: bool`
- `max_concurrency: int | None`
- `cost_hint: Literal["low", "medium", "high", "variable"]`
- `latency_hint: Literal["interactive", "batch", "slow"]`
- `input_schema: dict[str, Any]` (optional extended args)
- `output_schema: dict[str, Any]` (optional structured result)

Rationale: merges planner-facing metadata (today in `builtin` config + prompt text) with runtime capability metadata in one authoritative descriptor.

### 5.2 SubagentInvocationRequest

Fields:

- `request_id: str`
- `subagent_id: str`
- `message: str`
- `context: dict[str, Any]`
- `execution_mode: Literal["oneshot", "session"]`
- `timeout_s: float | None`
- `cancellation_token: str | None`
- `policy_overrides: dict[str, Any]`
- `metadata: dict[str, Any]`

### 5.3 SubagentInvocationResult

Fields:

- `request_id: str`
- `subagent_id: str`
- `success: bool`
- `final_text: str`
- `structured_output: dict[str, Any] | None`
- `artifacts: list[dict[str, Any]]`
- `usage: dict[str, Any] | None` (tokens/time/cost)
- `error_code: str | None`
- `error_message: str | None`
- `metadata: dict[str, Any]`

### 5.4 SubagentProgressEvent

Normalized events:

- `SUBAGENT_START`
- `SUBAGENT_PROGRESS`
- `SUBAGENT_TOOL_CALL`
- `SUBAGENT_TOOL_RESULT`
- `SUBAGENT_WARNING`
- `SUBAGENT_ERROR`
- `SUBAGENT_END`

Common envelope fields:

- `event_id`, `timestamp`, `request_id`, `subagent_id`, `event_type`, `summary`, `detail`, `payload`

### 5.5 BaseSubagentRuntime (Protocol)

```python
class BaseSubagentRuntime(Protocol):
    descriptor: SubagentDescriptor

    async def initialize(self) -> None: ...
    async def health(self) -> bool: ...
    async def invoke(self, request: SubagentInvocationRequest) -> SubagentInvocationResult: ...
    async def invoke_stream(self, request: SubagentInvocationRequest) -> AsyncGenerator[SubagentProgressEvent, None]: ...
    async def cancel(self, request_id: str) -> bool: ...
    async def cleanup(self) -> None: ...
```

Rules:

- `invoke` MUST be supported.
- `invoke_stream` MAY degrade to synthesized start/end when not natively streaming.
- `cancel` SHOULD be best-effort for external backends.
- `cleanup` MUST be idempotent.

## 6. Integration with CapabilityRegistry

### 6.1 Recommended Positioning

Subagents remain first-class capabilities (`CapabilityType.AGENT`) while gaining a dedicated runtime contract:

- `SubagentProvider` wraps runtime factory + descriptor.
- `CapabilityRegistry` stores and discovers providers.
- `SubagentManager` resolves provider by `subagent_id` and executes runtime contract.

This preserves existing registry architecture and avoids introducing a disconnected parallel registry.

### 6.2 Adapter Mapping

Current providers map as:

- `AgentCapabilityProvider` -> `InProcSubagentRuntimeAdapter`
- `BuiltInAgentCapabilityProvider` -> `BuiltinSubagentRuntimeAdapter`
- `CliAgentCapabilityProvider` -> `CliSubagentRuntimeAdapter`

All adapters return the same request/result/event shapes.

## 7. Routing and Policy Contract

Introduce `SubagentRoutingPolicy`:

- `allow_auto_routing: bool`
- `requires_explicit_command: bool`
- `allowed_parent_agent_types: list[str]`
- `max_depth: int`
- `permission_profile: str`
- `fallback_toolkits: list[str]`
- `deny_reasons: list[str]`

Enforcement layers:

1. **Planner hinting**: planner sees routeability metadata.
2. **Router validation**: blocks invalid automatic routing.
3. **Runtime enforcement**: hard fails if policy violated.

This formalizes todayâ€™s implicit behavior (e.g., explicit-only subagents).

## 8. Error Model

Standard error codes:

- `SUBAGENT_NOT_FOUND`
- `SUBAGENT_UNHEALTHY`
- `SUBAGENT_POLICY_DENIED`
- `SUBAGENT_TIMEOUT`
- `SUBAGENT_CANCELLED`
- `SUBAGENT_BACKEND_ERROR`
- `SUBAGENT_PROTOCOL_ERROR`

Guidelines:

- Parent agents should receive machine-readable `error_code` and user-safe message.
- Streaming failures emit `SUBAGENT_ERROR` then `SUBAGENT_END`.
- Partial outputs may be returned with `success=false` if useful.

---

## 9. Concurrency and Lifecycle

- Subagent runtimes are lazily initialized.
- Concurrency is controlled via descriptor + manager-level semaphore.
- Parent shutdown must trigger deterministic cleanup of active runtimes.
- Runtime health checks are optional on hot path; required for periodic monitoring.

## 10. Compatibility and Migration Plan

### Phase 1 (Non-breaking)

- Add new interfaces in `core/agent/subagent/`.
- Build adapters around existing providers.
- Keep current `SubagentAction` inputs, but route internally through `SubagentManager`.

### Phase 2 (Convergence)

- Replace string-action branching in `subagent_node` with typed request dispatch.
- Normalize event emission pipeline to `SubagentProgressEvent`.

### Phase 3 (Cleanup)

- Deprecate legacy special-case branches.
- Use descriptor/policy metadata directly in planner prompt construction.

## 11. Alternatives Considered

### Alternative A: Keep current provider model, no new subagent runtime interface

**Pros**
- Zero migration cost
- No new abstractions

**Cons**
- Continued subtype-specific branching
- Harder onboarding of new backends
- Inconsistent lifecycle/streaming semantics

**Verdict**: Rejected; insufficient for long-term extensibility.

### Alternative B: Separate SubagentRegistry independent from CapabilityRegistry

**Pros**
- Clear conceptual boundary for subagents
- Could simplify subagent-specific discovery APIs

**Cons**
- Duplicated registration/discovery logic
- Increased cognitive overhead
- Divergence risk between tool and agent ecosystems

**Verdict**: Rejected for now; integration cost outweighs benefits.

### Alternative C (Recommended): CapabilityRegistry + standardized SubagentRuntime contract

**Pros**
- Reuses proven registry foundation
- Unified interface with explicit runtime semantics
- Smooth incremental migration path

**Cons**
- Requires adapter layer and temporary dual path

**Verdict**: Accepted.

### Alternative D: Treat every subagent as a tool only

**Pros**
- One invocation model in prompts

**Cons**
- Blurs ontological boundary from RFC-1005
- Loses agent-level lifecycle/streaming semantics
- Poor fit for multi-step delegated cognition

**Verdict**: Rejected; conflicts with RFC-1005 design intent.

## 12. Open Questions

1. Should `SubagentInvocationResult.structured_output` be constrained via typed schemas per subagent category?
2. Should manager-level retries be opt-in per descriptor and error class?
3. Should remote subagents be required to implement event replay for resilience?
4. Should subagent usage accounting be merged into a cross-agent cost ledger RFC?

---

## 13. Security and Safety Considerations

- Enforce permission profiles before invocation (especially CLI/remote backends).
- Apply depth and recursion limits globally to prevent uncontrolled delegation.
- Sanitize streamed details for UI surfaces where required.
- Keep explicit-command-only subagents protected from automatic LLM routing.

---

## 14. Success Criteria

- Adding a new subagent backend requires only:
  1) Descriptor declaration
  2) Runtime adapter implementation
  3) Provider registration
- `subagent_node` no longer contains backend-specific branches.
- Streaming behavior is consistent across subagent types.
- Planner receives stable machine-readable routing metadata.
