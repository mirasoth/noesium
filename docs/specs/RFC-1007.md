## 1. Abstract

This RFC defines the layered architecture for the Noesium cognitive agentic framework, establishing clear separation between framework primitives (`noesium.core`), built-in toolkits (`noesium.toolkits`), reusable subagent implementations (`noesium.subagents`), and the application layer (`noesium.noeagent`). The key principle is that lower layers MUST NOT import from higher layers, ensuring a clean, reusable, and testable framework architecture.

---

## 2. Scope and Non-Goals

### 2.1 Scope

This RFC defines:

* The four-layer architecture: core, toolkits, subagents, noeagent
* Dependency direction rules between layers
* Import restrictions to enforce layering
* Configuration hierarchy and naming conventions
* Public API surface for each layer

### 2.2 Non-Goals

This RFC does **not** define:

* Specific implementation details of each layer (covered by other RFCs)
* Migration path for existing code (implementation detail)
* External package dependencies (covered by pyproject.toml)

---

## 3. Background & Motivation

The Noesium framework has evolved organically, leading to circular dependencies where framework-level code (in `noesium.core`) imports from application-level code (in `noesium.noeagent`). This violates fundamental software architecture principles and creates several problems:

1. **Testability**: Cannot test framework components in isolation
2. **Reusability**: Cannot use core/toolkits/subagents independently
3. **Maintainability**: Changes propagate unpredictably across layers
4. **Clarity**: Developers cannot easily understand the codebase structure

This RFC establishes a clean layered architecture to resolve these issues.

---

## 4. Design Principles

1. **Dependency Direction**: `core` ã `toolkits` ã `subagents` ã `noeagent`
2. **No Upstream Knowledge**: Lower layers (`core`, `toolkits`, `subagents`) CANNOT import from higher layers (`noeagent`)
3. **Interface Segregation**: Each layer exposes clear public APIs via `__init__.py`
4. **Configuration Hierarchy**: Core provides base configs, application layers extend

### 5.2 Layer Details

| Layer | Package | Purpose | Dependencies | Knows About NoeAgent |
|-------|---------|---------|--------------|---------------------|
| Core | `noesium.core` | Framework primitives (agents, tools, events, memory, LLM) | None (external only) | NO |
| Toolkits | `noesium.toolkits` | Built-in tool implementations | `noesium.core` | NO |
| Subagents | `noesium.subagents` | Reusable agent implementations | `noesium.core`, `noesium.toolkits` | NO |
| Application | `noesium.noeagent` | Complete agent application | All above | YES (it IS noeagent) |

### 5.3 Layer Contents

#### 5.3.1 Core Layer (`noesium.core`)

Framework primitives with zero knowledge of any application:

* `noesium.core.agent` - Base agent classes (BaseAgent, BaseGraphicAgent, BaseHitlAgent, BaseResearcher)
* `noesium.core.toolify` - Toolkit framework, registry, executor
* `noesium.core.event` - Event system, event store, progress events
* `noesium.core.memory` - Memory providers and managers
* `noesium.core.llm` - LLM client abstraction
* `noesium.core.capability` - Capability registry
* `noesium.core.config` - Framework configuration (renamed to FrameworkConfig)

#### 5.3.2 Toolkits Layer (`noesium.toolkits`)

Built-in tool implementations that extend core:

* Individual toolkit files (bash_toolkit.py, web_search_toolkit.py, etc.)
* Each toolkit inherits from `noesium.core.toolify.BaseToolkit` or `AsyncBaseToolkit`
* Registered via `@register_toolkit` decorator

#### 5.3.3 Subagents Layer (`noesium.subagents`)

Reusable subagent implementations:

* `noesium.subagents.askura` - General-purpose conversation agent
* `noesium.subagents.tacitus` - Research agent
* `noesium.subagents.bu` - Browser automation agent

Each subagent inherits from core base classes and may use toolkits.

#### 5.3.4 Application Layer (`noesium.noeagent`)

Complete application implementation:

* `noesium.noeagent.agent` - NoeAgent implementation
* `noesium.noeagent.config` - NoeConfig (extends framework config)
* `noesium.noeagent.tui` - Terminal UI
* `noesium.noeagent.cli_adapter` - CLI interface

## 6. Import Rules

### 6.1 Allowed Imports

```python
# CORRECT: Lower layers import from upper layers
from noesium.core import BaseAgent              # toolkits, subagents, noeagent can do this
from noesium.core.event import ProgressEvent    # toolkits, subagents, noeagent can do this
from noesium.toolkits import get_toolkit        # subagents, noeagent can do this
from noesium.subagents import AskuraAgent       # noeagent can do this
```

### 6.2 Forbidden Imports

```python
# WRONG: Lower layers importing from higher layers
from noesium.noeagent import NoeAgent           # core, toolkits, subagents CANNOT do this
from noesium.noeagent.progress import ProgressEvent  # core, toolkits, subagents CANNOT do this
from noesium.noeagent.config import NoeConfig   # core, toolkits, subagents CANNOT do this
```

### 6.3 Verification

The following command should return zero matches:

```bash
grep -r "from noesium.noeagent" noesium/core noesium/toolkits noesium/subagents
grep -r "import noesium.noeagent" noesium/core noesium/toolkits noesium/subagents
```

## 7. Configuration Hierarchy

### 7.1 FrameworkConfig (Core)

Located in `noesium/core/config.py`:

```python
class FrameworkConfig(BaseModel):
    """Framework-level configuration shared by all agents."""
    llm: LLMConfig
    tools: ToolsConfig
    memory: MemoryConfig
    logging: LoggingConfig
    # ... framework-wide settings
```

### 7.2 NoeConfig (Application)

Located in `noesium/noeagent/config.py`:

```python
class NoeConfig(BaseModel):
    """NoeAgent-specific configuration extending framework config."""
    
    @classmethod
    def from_global_config(cls) -> "NoeConfig":
        """Load from FrameworkConfig and add NoeAgent-specific settings."""
        from noesium.core.config import load_config
        global_config = load_config()
        # ... transform to NoeConfig
```

## 12. Conclusion

This RFC establishes a clean layered architecture for Noesium where:

1. **Core** provides framework primitives with no application knowledge
2. **Toolkits** extend core with built-in tools
3. **Subagents** build on core and toolkits for reusable agent implementations
4. **NoeAgent** is the application layer that integrates everything

This separation enables independent testing, reusability, and maintainability while providing clear boundaries for future development.

> **A framework should be usable without knowing about any application built on top of it.**
