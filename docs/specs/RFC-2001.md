## 1. Abstract

This RFC defines the concrete Python implementation design for the Noesium core framework. It translates the abstract architectural principles from [RFC-0001](RFC-0001.md) through [RFC-1003](RFC-1003.md) into specific module structures, class hierarchies, interface contracts, data structures, and configuration patterns. The design targets Python 3.11+ and uses LangGraph as the graph execution substrate, bubus as the in-process event bus, and Pydantic for data modeling.

---

## 2. Scope and Non-Goals

### 2.1 Scope

This RFC defines:

* Module layout and package structure for `noesium/core/`
* Base class hierarchies and abstract interfaces
* Event system implementation (ProgressEvent)
* Memory hierarchy implementation (ephemeral, memu, semantic)
* Capability registry implementation
* Tool system (Toolify) integration
* LLM client abstraction layer
* Configuration management and dependency injection patterns
* Tracing and observability integration

### 2.2 Non-Goals

This RFC does **not** define:

* Specific agent implementations (see [RFC-1002](RFC-1002.md))
* Individual toolkit implementations
* Deployment topology or infrastructure provisioning
* External service integrations (databases, message queues)
* CLI or user-facing application interfaces

---

## 3. Design Principles

1. **Simplicity First**: Common patterns SHOULD work with minimal configuration; advanced features activated through explicit opt-in.
2. **Type Safety**: All public interfaces MUST use Pydantic models or typed protocols; runtime type checking via Pydantic validation.
3. **Async First**: Core interfaces MUST be async; synchronous wrappers provided as convenience.
4. **Testability**: All components MUST be injectable and mockable; no hidden global state in core interfaces.

## 5. Event System

### 5.1 ProgressEvent

The primary event type for TUI streaming:

```python
@dataclass
class ProgressEvent:
    type: ProgressEventType
    message: str
    metadata: dict[str, Any] = field(default_factory=dict)
    timestamp: datetime = field(default_factory=lambda: datetime.now(tz=timezone.utc))


class ProgressEventType(str, Enum):
    THINKING = "thinking"
    TOOL_CALL = "tool_call"
    TOOL_RESULT = "tool_result"
    SUBAGENT_START = "subagent_start"
    SUBAGENT_PROGRESS = "subagent_progress"
    SUBAGENT_END = "subagent_end"
    TEXT = "text"
    ERROR = "error"
    COMPLETE = "complete"
```

### 5.2 Event Bus Integration

The system uses **bubus** for event dispatch:

```python
from bubus import EventBus

bus = EventBus()
bus.subscribe(ProgressEvent, handler)
await bus.publish(event)
```

### 8.1 Hierarchy

```
├── Token tracking, LLM client management

│   │   └── Human-in-the-loop patterns
│   │
│   └── TacitusAgent (research-style)
│       ├── Research workflow patterns
│       └── Citation tracking
```

### 8.2 BaseAgent

```python
class BaseAgent(ABC):
    """Base agent with LLM client management."""

    def __init__(
        self,
        agent_id: str | None = None,
        llm_provider: str | None = None,
        llm_model: str | None = None,
    ):
        self._agent_id = agent_id or str(uuid7())
        self._producer = {
            "agent_id": self._agent_id,
            "agent_type": self.__class__.__name__,
        }
        ...

    @abstractmethod
    async def run(self, input_data: str) -> str: ...
```

### 8.3 BaseGraphicAgent

```python
class BaseGraphicAgent(BaseAgent):
    """LangGraph-based agent with capability support."""

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self._capabilities: list[Capability] = []
        ...

    @abstractmethod
    def _build_graph(self) -> StateGraph: ...

    def declare_capability(self, capability: Capability) -> None:
        """Declare a capability this agent provides."""
        self._capabilities.append(capability)
```

## 15. Conclusion

RFC-2001 defines the concrete Python implementation design for the Noesium core framework. The framework provides:

* Lightweight event streaming via ProgressEvent
* Memory hierarchy with ephemeral, memu (persistent), and semantic layers
* Provider-based capability registry with local changelog
* CognitiveContext for cross-turn state continuity
* LangGraph integration for graph-based agent execution
* Toolify system for tool management

> The core framework provides the execution substrate; agents provide the cognitive logic.
