## 1. Abstract

This RFC defines the **Autonomous Architecture** for NoeAgent. The goal is to evolve NoeAgent from an interactive agent runtime into a **24/7 autonomous cognitive system** capable of long-running tasks, continuous reasoning, and goal-driven behavior.

The design intentionally minimizes architectural complexity while preserving the core primitives required for autonomy.

The architecture introduces two new components:

* Goal Engine (detailed in [RFC-1006](RFC-1006.md))
* Cognitive Loop

All other components reuse the existing NoeAgent architecture.

The resulting system forms a minimal cognitive architecture composed of five core components.

---

## 2. Scope and Non-Goals

### 2.1 Scope

This RFC defines:
* High-level autonomous architecture for NoeAgent
* Two new core components: Goal Engine and Cognitive Loop
* Integration with existing systems (Agent Kernel, Capability System, Memory System)
* Execution flow for autonomous operation
* Architectural principles and design goals

### 2.2 Non-Goals

This RFC does **not** define:
* Detailed implementation of Goal Engine (see [RFC-1006](RFC-1006.md))
* Detailed event system design (see [RFC-1007](RFC-1007.md))
* Multi-agent orchestration mechanisms
* Distributed agent coordination
* Learning and adaptation mechanisms

## 3. Background & Motivation

NoeAgent currently operates as an interactive agent runtime, responding to user requests in a request-response pattern. While effective for interactive use cases, this model cannot support autonomous, long-running cognitive behavior required for advanced AI applications such as continuous research, self-directed learning, and persistent task execution.

The autonomous architecture addresses this limitation by introducing goal-driven execution and continuous cognitive loops, enabling NoeAgent to operate independently over extended time periods while maintaining deterministic behavior and system simplicity.

---

## 4. Design Goals

The architecture is designed according to the following principles:

1. **Minimalism**

   * Minimize the number of core system components.

2. **Deterministic Execution**

   * Preserve the deterministic execution model.

3. **Continuous Operation**

   * Enable 24/7 autonomous execution.

4. **Goal-Driven Behavior**

   * Transition from user-driven execution to goal-driven cognition.

5. **Reuse Existing Systems**

   * Reuse the Capability Registry and Memory Projection model.

6. **Avoid System Explosion**

   * Avoid multi-agent orchestration, workflow engines, or overly complex planning layers.

---

## 5. Architecture Overview

The NoeAgent Autonomous Architecture consists of five core components:

1. Goal Engine
2. Cognitive Loop
3. Agent Kernel
4. Capability System
5. Memory System

High-level architecture:

```

                +----------------------+
                |      Goal Engine     |
                +----------+-----------+
                           |
                           v
                +----------+-----------+
                |     Cognitive Loop   |
                +----------+-----------+
                           |
                           v
                +----------+-----------+
                |     Agent Kernel     |
                +----------+-----------+
                           |
        +------------------+------------------+
        |                                     |
        v                                     v
 Capability System                      Memory System
 (Tools / Subagents)                     (Projection)

```

This architecture forms the minimal runtime required for an autonomous cognitive agent.

## 6. Goal Engine

The Goal Engine is responsible for managing agent objectives.

It does **not perform reasoning**. Instead, it manages goal lifecycle and scheduling.

### Responsibilities

* Goal creation
* Goal prioritization
* Goal scheduling
* Goal lifecycle management

### Goal Structure

```
Goal
 ├ id
 ├ description
 ├ priority
 ├ status
 ├ created_at
 └ parent_goal
```

### Goal Status

```
pending
active
blocked
completed
failed
```

### Goal Sources

Goals may originate from:

* user input
* memory triggers
* external events
* self-generated goals

Example goals:

* monitor new research papers
* review project repository issues
* analyze new datasets

### Goal Queue

The Goal Engine maintains a priority queue of active goals.

Scheduling strategy:

```
highest_priority_first
```

## 7. Cognitive Loop

The Cognitive Loop is the continuous runtime loop that drives autonomous execution.

The loop repeatedly selects a goal, generates reasoning, performs actions, and updates memory.

### Core Loop

```
while true:

    goal = goal_engine.next_goal()

    context = memory.project(goal)

    decision = agent_kernel.step(goal, context)

    observation = execute(decision)

    memory.update(observation)
```

### Loop Tick

Each iteration of the cognitive loop is called a **tick**.

Typical tick duration:

* 5–30 seconds

The loop runs continuously, enabling persistent long-running cognition.

### Input Sources

The loop may incorporate inputs from multiple sources:

* goal engine
* memory triggers
* external events
* tool observations

---

## 8. Agent Kernel

The Agent Kernel is the reasoning engine of NoeAgent.

It performs cognitive reasoning and produces executable decisions.

### Responsibilities

* reasoning
* planning
* tool selection
* subagent invocation
* goal updates

### Kernel Interface

```
step(goal, context) -> decision
```

### Decision Types

Possible decisions include:

* tool_call
* subagent_call
* memory_update
* goal_update
* finish_goal

The kernel may internally perform multi-step reasoning (e.g., ReAct-style reasoning) but exposes a single step interface to the Cognitive Loop.

---

## 9. Capability System

The Capability System provides executable functionality to the agent.

Capabilities unify multiple execution mechanisms under a single abstraction.

### Capability Types

```
Capability
 ├ Tool
 ├ Subagent
 └ Service
```

Examples:

* browser.search
* github.read
* filesystem.write
* subagent.researcher
* subagent.coder

Capabilities are discovered through the Capability Registry defined in [RFC-1003](RFC-1003.md).

Execution flow:

```
decision -> capability -> execution -> observation
```

Observations are returned to the Cognitive Loop and stored in memory.

---

## 10. Memory System

The Memory System stores persistent knowledge and agent state.

It uses the Projection Model defined in [RFC-1002](RFC-1002.md).

### Projection Model

```
Global Memory
      |
      v
Projection(goal)
      |
      v
Agent Context
```

Memory provides the working context used during reasoning.

### Memory Types

Typical memory categories include:

* episodic memory
* semantic memory
* goal memory
* execution logs

Memory is updated after each cognitive loop iteration.

---

## 11. Execution Flow

The full autonomous execution pipeline:

```
Goal Engine
      |
      v
Cognitive Loop
      |
      v
Agent Kernel
      |
      v
Capability Execution
      |
      v
Observation
      |
      v
Memory Update
```

This pipeline repeats indefinitely during system runtime.

---

## 12. Minimal Cognitive Architecture

The architecture forms a minimal cognitive system consisting of:

* motivation (Goal Engine)
* cognition (Agent Kernel)
* action (Capability System)
* memory (Memory System)
* runtime loop (Cognitive Loop)

This provides the minimal set of primitives required for an autonomous AI system.

---

## 13. Non-Goals

The following are explicitly **not included** in this architecture:

* multi-agent swarm architectures
* complex workflow orchestration
* hierarchical planning frameworks
* distributed agent clusters

These may be considered future extensions but are intentionally excluded to preserve system simplicity.

---

## 14. Future Extensions

Possible future extensions include:

* world model integration
* event-driven goal generation
* distributed capability execution
* persistent agent learning

These extensions are outside the scope of this RFC.

---

## 15. Conclusion

This RFC introduces a minimal architecture that transforms NoeAgent into a continuously operating autonomous agent.

By adding only two new components (Goal Engine and Cognitive Loop), the system evolves from an interactive tool runner into a goal-driven cognitive runtime while preserving the deterministic and modular properties of the original design.
