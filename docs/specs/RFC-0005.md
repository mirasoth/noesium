## 1. Abstract

This RFC defines the **Capability Registry and Discovery Protocol (CRDP)** for Noesium.

The registry enables:

* Deterministic declaration of agent capabilities
* Typed discovery across agents
* Versioned capability negotiation
* Runtime routing without tight coupling
* Multi-agent composability

The protocol ensures that:

* Capabilities are declared via events
* Discovery is projection-based
* Resolution is deterministic
* Invocation is event-driven
* Agents remain loosely coupled

This forms the **contract layer** between autonomous agents.

---

## 2. Problem Statement

In a multi-agent system:

* Agents need to discover other agents.
* Capabilities evolve over time.
* Invocation must be deterministic and auditable.
* Agents must remain replaceable.

Ad-hoc routing leads to:

* Tight coupling
* Hidden dependencies
* Runtime ambiguity
* Non-deterministic behavior

Noesium requires a formal capability model.

---

## 3. Design Principles

1. Capabilities are declared, not inferred.
2. Capability metadata is immutable and versioned.
3. Discovery is projection-based and replayable.
4. Invocation is event-mediated.
5. Resolution must be deterministic inside the Kernel.
6. Registry state must be reconstructable from events.

---

## 4. Conceptual Model

### 4.1 Capability Definition

A capability is a typed contract:

```
Capability C = {
    id: string,
    version: semver,
    input_schema: JSONSchema,
    output_schema: JSONSchema,
    side_effects: enum,
    latency_class: enum,
    determinism_class: enum,
    capability_type: CapabilityType
}
```

Where:

* `determinism_class ∈ {deterministic, nondeterministic}`
* `side_effects ∈ {pure, idempotent, external}`

---

### 4.2 Capability Type Taxonomy

Every registered capability belongs to exactly one type:

```
CapabilityType := {
    InProcessTool,
    RemoteTool,
    Subagent,
    ExternalCliSubagent
}
```

| Type | Nature | State | Lifecycle | Invocation |
|------|--------|-------|-----------|------------|
| `InProcessTool` | Stateless callable | None | Per-call | Synchronous / async function |
| `RemoteTool` | Stateless remote callable | None | Per-call | MCP, HTTP, RPC |
| `Subagent` | Stateful cognitive worker | Session-scoped | Managed by orchestrator | In-process async |
| `ExternalCliSubagent` | Stateful CLI daemon | Session-scoped, persistent | Managed by adapter | stdin/stdout pipes |

**Design invariants:**

* **Tools** (`InProcessTool`, `RemoteTool`) are stateless, atomic, cheap, and fully controlled by the caller. They must be referentially transparent when possible.
* **Subagents** (`Subagent`, `ExternalCliSubagent`) are stateful, autonomous within boundaries, long-lived, and own partial reasoning space. They are intentionally not referentially transparent.

A tool extends **capability**. A subagent extends **cognition**.

---

### 4.3 Agent Capability Set

Each agent A declares:

```
Capabilities_A = {C1, C2, C3}
```

Declared through event emission:

```
CapabilityRegistered
CapabilityDeprecated
CapabilityUpdated
```

---

### 4.4 Registry as Projection

The registry is not a database table.

It is a projection:

```
RegistryState = fold(P_registry, ∅, EventStream)
```

Where:

```
P_registry : (State, Event) → State
```

Constraints:

* Pure function
* Deterministic
* No IO
* Replayable

Aligned with RFC-0003 and RFC-0004.

## 6. Discovery Protocol

### 6.1 Deterministic Query Model

Discovery occurs through deterministic projection queries:

```
Find(CapabilityID, VersionRange)
FindByTag(Tag)
FindByDeterminismClass(Class)
FindByCapabilityType(CapabilityType)
```

These operate on `RegistryState`, not on external services.

---

### 6.2 Resolution Algorithm

When resolving a capability:

1. Filter by ID
2. Filter by version compatibility
3. Filter by capability type (if constrained)
4. Filter by health status
5. Apply deterministic ranking
6. Select target agent

Ranking must not depend on:

* Randomness
* Wall clock
* External metrics

Deterministic ranking may use:

* Static priority
* Registration order
* Explicit weight field

---

## 7. Invocation Flow

### 7.1 Tool Invocation (InProcessTool, RemoteTool)

1. Caller emits `capability.requested`
2. Kernel resolves deterministically
3. Kernel emits `capability.invoked`
4. Tool executes atomically
5. Result returned as `capability.completed`

No persistent state. No lifecycle management.

### 7.2 Subagent Invocation (Subagent, ExternalCliSubagent)

1. Caller emits `capability.requested`
2. Kernel resolves deterministically
3. Kernel emits `capability.invoked`
4. Subagent receives task via adapter
5. Subagent may execute multi-step internal loop
6. Subagent emits `capability.completed`

Subagent invocation may involve session creation, task queuing, and lifecycle transitions.

All invocations are event-mediated. No direct method calls.

---

## 8. Health & Liveness Model

Health is projection-based.

Agents emit:

```
agent.heartbeat
agent.unavailable
agent.recovered
```

Registry projection maintains:

```
HealthState[agent_id]
```

Resolution excludes unhealthy agents.

### 8.1 Subagent Liveness

Subagent types (`Subagent`, `ExternalCliSubagent`) require mandatory liveness monitoring:

* Periodic heartbeat validation
* Automatic health status transitions
* Integration with lifecycle model (§10)

Tool types do not require persistent health monitoring.

To avoid collision:

Examples:

```

```

## 10. Persistent Subagent Architecture

### 10.1 Motivation

Certain agent capabilities are provided by external CLI-based agents that:

* Maintain their own internal memory and session state
* Implement proprietary planning and execution strategies
* Provide interactive, incremental reasoning loops

Spawning them as short-lived subprocesses per request results in loss of session state, increased startup latency, and poor multi-step task continuity. Therefore, these agents are treated as **persistent CLI daemons**.

### 10.2 Subagent Adapter Interface

A generic adapter layer manages external subagent processes:

```
SubagentAdapter
  ├── spawn()           → SubagentHandle
  ├── send(message)     → RequestID
  ├── receive()         → Response
  ├── health_check()    → HealthStatus
  ├── restart()         → SubagentHandle
  └── terminate()       → void
```

Responsibilities:

* Launch subagent process (in-process or daemon mode)
* Maintain persistent communication channel
* Serialize/deserialize structured messages
* Monitor liveness

### 10.3 Communication Protocol

Communication occurs over structured channels:

* **In-process subagents**: Direct async method calls
* **External CLI subagents**: stdin/stdout streams with JSON envelope

Envelope structure:

```json
{
  "request_id": "uuid",
  "task": "description",
  "payload": "...",
  "context": {...}
}
```

The adapter guarantees:

* Request-response correlation
* Timeout handling
* Partial streaming support (optional)

### 10.4 Session Multiplexing

Each subagent instance corresponds to one logical session.

The orchestrator maintains:

```
subagent_sessions: Dict[SessionID, SubagentHandle]
```

Session policies:

* One daemon per user session (default)
* Optional shared daemon pool (future extension)
* Explicit session termination required

### 10.5 Capability Registration for Subagents

`ExternalCliSubagent` capabilities must additionally declare:

* Supported task types
* `stateful: true`
* Concurrency model (serial, limited-parallel)
* Resource cost hints (memory, CPU class)

## 11. Subagent Lifecycle Model

### 11.1 States

```
CREATED → RUNNING → BUSY → IDLE → TERMINATED
```

State transitions:

* `CREATED → RUNNING`: Adapter calls `spawn()`
* `RUNNING → BUSY`: Task dispatched
* `BUSY → IDLE`: Task completed
* `IDLE → BUSY`: New task dispatched
* `IDLE → TERMINATED`: Explicit termination or timeout
* `BUSY → TERMINATED`: Forced termination (timeout/crash)

### 11.2 Failure Recovery

If a subagent becomes unresponsive:

1. Detect via `health_check()`
2. Attempt automatic restart via `restart()`
3. Optionally restore session context (best effort)
4. Escalate failure to orchestrator if restart fails

### 11.3 Scheduling

Subagents are treated as heavyweight workers:

* Avoid spawning per request; reuse active sessions
* Queue tasks if subagent is BUSY
* Enforce concurrency caps per user/session
* Apply resource limits (memory, CPU, time)

---

## 12. Version Negotiation

Semver rules apply:

* Major mismatch → incompatible
* Minor upgrade → compatible
* Patch upgrade → safe

Negotiation must be deterministic.

If no compatible version:

→ Emit `capability.unresolved`

---

## 13. Security Model

### 13.1 Access Control

Capabilities may declare:

```
required_roles
required_scopes
```

Kernel validates before invocation.

---

### 13.2 Sandboxing

Invocation isolation:

* Agent cannot mutate registry
* Agent cannot bypass event bus
* No direct state mutation

### 13.3 Subagent Security

External CLI subagents introduce additional security surface:

* May call other tools internally
* May access filesystem
* May run long tasks

Security model for subagents (sandbox isolation, resource quotas, filesystem scoping, network restrictions) is defined in separate implementation RFCs. This RFC establishes the architectural boundary only.

---

## 14. Distributed Registry Model

For multi-node deployments:

Option A: Global event log
Option B: Partitioned registry with CRDT merge

If partitioned:

```
RegistryState = CRDTMerge(RegistryNodeStates)
```

Must converge deterministically.

## 15. Failure Modes

| Failure                | Resolution               |
| ---------------------- | ------------------------ |
| Agent crash            | Mark unavailable         |
| Duplicate registration | Idempotent via event ID  |
| Version conflict       | Deterministic resolution |
| Stale snapshot         | Replay                   |
| Subagent crash         | Restart via adapter, escalate if repeated |
| Subagent timeout       | Force terminate, return error result |

## 16. Comparison: Tool vs Subagent Capabilities

| Dimension | Tool (InProcess/Remote) | Subagent (In-process/CLI) |
|-----------|------------------------|---------------------------|
| Nature | Function / procedure | Cognitive worker |
| Identity | Stateless capability | Stateful actor |
| Control | Caller retains full control | Delegated autonomy |
| State | None | Session-scoped, persistent |
| Temporal scope | Atomic, bounded | Long-lived, async |
| Failure impact | Single request | Session-level |
| Recovery | Retry call | Restart process |
| Scheduling | Spawn per call, no queuing | Reuse instance, task queue |
| Determinism | Deterministic (ideal) | May be nondeterministic |
| Security surface | Narrow, parameter validation | Broad, internal tool access |
| Observability | Log input/output | Reasoning trace, state snapshot, metrics |

**Cognitive boundary principle:**

* Tool call: "Do this operation for me."
* Subagent call: "Handle this problem domain for me."

The second implies delegation of **problem ownership**.

Future additions may include:

* Capability cost modeling
* QoS-aware resolution
* SLA metadata
* Dynamic load balancing
* Marketplace-style scoring
* Shared CLI daemon pools
* gRPC-based agent bridging

## 18. Comparison to Existing Systems

Traditional systems:

* Central registry (e.g., service mesh)
* Runtime RPC resolution
* Imperative routing

Noesium:

* Event-sourced registry
* Projection-based discovery
* Deterministic resolution
* Event-mediated invocation
* Fully replayable routing decisions
* Persistent subagent lifecycle management

---

## 19. Guarantees

This RFC guarantees:

* Loose coupling between agents
* Deterministic capability resolution
* Replayable routing
* Version safety
* Health-aware invocation
* Composable multi-agent scaling
* Persistent subagent session management
* Typed capability taxonomy (tool vs subagent)

## 21. Conclusion

RFC-0005 establishes the contract and discovery layer of Noesium.

It transforms:

Agent calls → Event-mediated contracts
Service lookup → Deterministic projection
Dynamic routing → Replayable resolution
External CLIs → Persistent subagent daemons

This enables a scalable, deterministic, auditable, and composable multi-agent ecosystem with first-class support for both stateless tools and stateful cognitive subagents.
