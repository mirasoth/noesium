## 1. Abstract

This RFC defines the **Autonomous Goal Engine** for NoeAgent. The Goal Engine provides the motivational layer of the autonomous architecture introduced in [RFC-1005](RFC-1005.md).

The Goal Engine is intentionally minimal. Its purpose is to manage goals and provide the Cognitive Loop with the next objective to execute.

The Goal Engine **does not perform reasoning**. All reasoning remains inside the Agent Kernel.

Core responsibilities:

* goal storage
* goal lifecycle management
* goal prioritization
* goal scheduling

---

## 2. Scope and Non-Goals

### 2.1 Scope

This RFC defines:
* Goal model and data structure
* Goal lifecycle and state transitions
* Goal storage mechanisms
* Goal queue and scheduling policies
* Integration with Cognitive Loop and Agent Kernel
* Goal sources and creation mechanisms

### 2.2 Non-Goals

This RFC does **not** define:
* Complex planning systems
* Hierarchical task networks
* Reinforcement learning policies
* Multi-agent negotiation
* Event-driven trigger mechanisms (see [RFC-1007](RFC-1007.md))

---

## 3. Background & Motivation

The autonomous architecture requires a mechanism to determine *what* the agent should work on at any given time. Unlike interactive agents that respond to user requests, autonomous agents must self-direct their activities based on objectives, priorities, and environmental conditions.

The Goal Engine provides this motivational layer by managing a queue of objectives and selecting the next goal for execution. By keeping the Goal Engine minimal and deterministic, the architecture preserves simplicity while enabling sophisticated autonomous behavior through the Agent Kernel's reasoning capabilities.

---

## 4. Design Principles

The Goal Engine follows several strict principles:

### Minimalism

The system should include the smallest possible set of mechanisms required to support autonomous behavior.

### Separation of Concerns

The Goal Engine manages *what to work on*. The Agent Kernel determines *how to accomplish it*.

### Deterministic Scheduling

Goal selection must be deterministic whenever possible to preserve reproducibility.

### Extensibility

Future mechanisms such as event-driven goals or learning-based prioritization may extend the system without modifying the core model.

---

## 5. Goal Model

A goal represents an objective the agent should attempt to complete.

### Goal Structure

```
Goal
 ├ id
 ├ description
 ├ priority
 ├ status
 ├ created_at
 ├ updated_at
 └ parent_goal
```

Optional metadata may include tags, deadlines, or origin.

### Goal Status

```
pending
active
blocked
completed
failed
```

State transitions are managed by the Goal Engine and the Agent Kernel.

---

## 6. Goal Storage

Goals are stored persistently within the Memory System.

Typical storage fields:

```
goal_id
description
priority
status
created_at
updated_at
parent_goal
```

Goals are part of the agent's long-term memory and may persist across runtime sessions.

---

## 7. Goal Sources

Goals may originate from several sources.

### User Goals

Goals explicitly created by a user.

Examples:

* review repository pull requests
* analyze research papers

### Memory-Triggered Goals

Goals generated when memory conditions are satisfied.

Examples:

* new research paper detected
* repository activity increased

### External Events

Goals triggered by external systems.

Examples:

* webhook events
* scheduled tasks

### Self-Generated Goals

The Agent Kernel may propose new goals during reasoning.

Example:

* after reading research papers, generate goal to summarize findings

---

## 8. Goal Queue

The Goal Engine maintains a **priority queue** of active goals.

Scheduling policy:

```
highest_priority_first
```

If multiple goals share the same priority, the system selects the oldest goal.

```
(priority DESC, created_at ASC)
```

This ensures deterministic scheduling.

---

## 9. Goal Lifecycle

Goals progress through a lifecycle.

```
create
  ↓
pending
  ↓
active
  ↓
completed | failed | blocked
```

Lifecycle transitions may be triggered by:

* Agent Kernel decisions
* system policies
* external events

---

## 10. Kernel Interaction

The Agent Kernel interacts with the Goal Engine through a minimal interface.

### Kernel API

```
create_goal(description, priority)
update_goal(goal_id, status)
complete_goal(goal_id)
fail_goal(goal_id)
```

The Cognitive Loop retrieves the next goal using:

```
next_goal()
```

---

## 11. Integration with Cognitive Loop

Within the autonomous runtime, the Goal Engine provides the next goal to execute.

Execution model:

```
while true:

    goal = goal_engine.next_goal()

    context = memory.project(goal)

    decision = agent_kernel.step(goal, context)

    execute(decision)

    update_memory()
```

Goal completion or updates are handled during execution.

---

## 12. Determinism and Stability

To maintain system stability, the Goal Engine follows deterministic rules:

* priority-based selection
* stable ordering
* explicit lifecycle transitions

This ensures reproducibility for debugging and auditing.

---

## 13. Non-Goals

The Goal Engine intentionally excludes the following features:

* complex planning systems
* hierarchical task networks
* reinforcement learning policies
* multi-agent negotiation

These mechanisms may exist inside the Agent Kernel but not in the Goal Engine.

---

## 14. Future Extensions

Possible extensions include:

* event-driven goal triggers
* goal aging and decay
* deadline-based prioritization
* learning-based goal scoring

These extensions should not modify the core goal model defined in this RFC.

---

## 15. Conclusion

The Autonomous Goal Engine provides the minimal motivational layer required for a continuously operating NoeAgent.

By limiting the Goal Engine to goal management and scheduling, the architecture remains simple, deterministic, and extensible while enabling true autonomous behavior.
