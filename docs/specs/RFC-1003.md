## 1. Abstract

This RFC defines the **Capability Registry and Discovery Protocol (CRDP)** for Noesium.

The registry enables:

* Deterministic declaration of agent capabilities
* Typed discovery across agents
* Versioned capability negotiation
* Runtime routing without tight coupling
* Multi-agent composability

The protocol ensures that:

* Capabilities are held as live **CapabilityProvider** objects in-memory (provider-first)
* Discovery is projection-based on the in-memory registry
* Resolution is deterministic and health-aware
* Invocation is **hybrid**: direct for tools, event-mediated for agents
* Events are emitted for **audit and observability**, not as the source of truth
* Registry state is reconstructable from events on cold-start (hybrid event sourcing)

This forms the **contract layer** between autonomous agents.

---

## 2. Problem Statement

In a multi-agent system:

* Agents need to discover other agents.
* Capabilities evolve over time.
* Invocation must be deterministic and auditable.
* Agents must remain replaceable.

Ad-hoc routing leads to:

* Tight coupling
* Hidden dependencies
* Runtime ambiguity
* Non-deterministic behavior

Noesium requires a formal capability model.

---

## 3. Design Principles

1. **Provider-first, not event-first**: The registry holds live `CapabilityProvider` objects in-memory. Events are emitted as a side-effect for audit and observability, not as the source of truth.
2. Capabilities are declared, not inferred.
3. Capability metadata is immutable and versioned.
4. Discovery operates on the in-memory registry; events support replay for cold-start.
5. Resolution must be deterministic and health-aware inside the Kernel.
6. Registry state must be reconstructable from events for audit and cold-start (`rebuild_from_events()`).

## 4. Conceptual Model

### 4.1 CapabilityDescriptor

A capability is described by a typed contract. `CapabilityDescriptor` replaces the old `Capability` model:

```
CapabilityDescriptor = {
    capability_id: string,
    version: semver,
    capability_type: CapabilityType,
    description: string,
    input_schema: JSONSchema,
    output_schema: JSONSchema,
    determinism: DeterminismClass,
    side_effects: SideEffectClass,
    latency: LatencyClass,
    tags: list[string]
}
```

Derived property:

* `stateful: bool` — `true` iff `capability_type ∈ {AGENT, CLI_AGENT}`

Where:

* `determinism ∈ {deterministic, stochastic, external}`
* `side_effects ∈ {pure, idempotent, effectful}`
* `latency ∈ {realtime, fast, batch}`

---

### 4.2 Capability Type Taxonomy

Every registered capability belongs to exactly one of five types:

```
CapabilityType := {
    TOOL,
    MCP_TOOL,
    SKILL,
    AGENT,
    CLI_AGENT
}
```

| Type | Nature | State | Lifecycle | Invocation |
|------|--------|-------|-----------|------------|
| `TOOL` | In-process toolkit tool, stateless | None | Per-call | Direct `provider.invoke()` |
| `MCP_TOOL` | Remote MCP tool, stateless | None | Per-call | Direct `provider.invoke()` |
| `SKILL` | Composite multi-tool skill, stateless | None | Per-call | Direct `provider.invoke()` |
| `AGENT` | In-process subagent, stateful | Session-scoped | Managed by orchestrator | Event-mediated |
| `CLI_AGENT` | External CLI daemon, stateful | Session-scoped, persistent | Managed by adapter | Event-mediated |

**Design invariants:**

* **Stateless types** (`TOOL`, `MCP_TOOL`, `SKILL`) are atomic, cheap, and fully controlled by the caller. They must be referentially transparent when possible.
* **Stateful types** (`AGENT`, `CLI_AGENT`) are autonomous within boundaries, long-lived, and own partial reasoning space. They are intentionally not referentially transparent.

A tool extends **capability**. A subagent extends **cognition**.

---

### 4.3 CapabilityProvider Protocol

All capability types implement a unified protocol:

```
CapabilityProvider = {
    descriptor: CapabilityDescriptor,
    invoke(**kwargs) -> Any,
    health() -> bool
}
```

* `descriptor` — immutable metadata
* `invoke(**kwargs)` — execute the capability
* `health()` — liveness check (stateless providers always return `True`; stateful providers report actual status)

---

### 4.4 Registry as Projection (Hybrid)

The registry is **provider-first** with **hybrid event sourcing**:

* **In-memory dict** — holds live `CapabilityProvider` objects for fast reads. This is the source of truth for discovery and resolution.
* **Append-only changelog** — `list[RegistryEvent]` for audit. Every registration, unregistration, and health change is recorded.
* **Async flush to EventStore** — when an `EventStore` is supplied, changelog entries are flushed asynchronously for persistence and cross-process replay.
* **Cold-start** — `rebuild_from_events()` reconstructs the changelog from a persisted event stream. Live providers must be re-registered by agents on startup.

```
RegistryState = in_memory_providers + changelog
changelog = fold(P_registry, [], EventStream)  # for audit / replay
```

Aligned with RFC-0004.

## 5. Event Schema

All events follow RFC-0002 envelope.

**Capability events are for audit and observability, not for routing.** The registry does not derive its live state from events; it derives it from `register()` / `unregister()` calls. Events enable:

* Audit trail
* Cross-process replay
* Cold-start changelog reconstruction

### 5.1 CapabilityRegistered

```
{
  "event_type": "capability.registered",
  "agent_id": "agent-X",
  "capability_id": "summarize.text",
  "capability_type": "TOOL",
  "version": "1.2.0",
  "input_schema": {...},
  "output_schema": {...},
  "metadata": {...}
}
```

---

### 5.2 CapabilityDeprecated

```
{
  "event_type": "capability.deprecated",
  "capability_id": "...",
  "reason": "..."
}
```

---

### 5.3 CapabilityInvoked / CapabilityCompleted

```
{
  "event_type": "capability.invoked",
  "caller_agent_id": "...",
  "target_agent_id": "...",
  "capability_id": "...",
  "correlation_id": "..."
}
```

These events are emitted for observability. For tools, the ToolExecutor emits tool-level events. For agents, the event-mediated flow emits `capability.requested` → `capability.invoked` → `capability.completed`.

## 6. Discovery Protocol

### 6.1 Unified CapabilityRegistry

A single `CapabilityRegistry` replaces the former ToolRegistry, DiscoveryService, and DeterministicResolver.

**Registration:**

* `register(provider)` — add a provider
* `unregister(capability_id [, version])` — remove providers

**Discovery:**

* `find(capability_id [, version])` — find providers by ID
* `find_by_type(capability_type)` — filter by type
* `find_by_tag(tag)` — filter by tag
* `query(CapabilityQuery)` — structured query across all dimensions

**Resolution:**

* `resolve(capability_id [, version])` — deterministic, health-aware resolution; returns first healthy match by registration order
* `get_by_name(name)` — O(1) lookup by capability_id (last registered wins)

**Introspection:**

* `list_providers([capability_type])` — all providers
* `list_descriptors()` — all descriptors

**Observer pattern:**

* `add_listener(callback)` — notified on every registry mutation (registration, unregistration, health change)

**Changelog:**

* Append-only `list[RegistryEvent]` for audit

---

### 6.2 Resolution Algorithm

When resolving a capability:

1. Filter by ID
2. Filter by version compatibility
3. Filter by capability type (if constrained)
4. Filter by health status (use cached health for stateful providers)
5. Apply deterministic ranking (registration order)
6. Select target provider

Ranking must not depend on:

* Randomness
* Wall clock
* External metrics

## 7. Invocation Flow

### 7.1 Hybrid Invocation (Option C)

**Tools (TOOL, MCP_TOOL, SKILL):** Direct invocation via `provider.invoke()`. The ToolExecutor emits tool-level events for observability. No event-mediated routing.

**Agents (AGENT, CLI_AGENT):** Event-mediated flow:

1. Caller emits `capability.requested`
2. Kernel resolves deterministically
3. Kernel emits `capability.invoked`
4. `provider.invoke()` is called
5. Subagent executes (possibly multi-step)
6. Kernel emits `capability.completed`

---

### 7.2 Tool Invocation (TOOL, MCP_TOOL, SKILL)

1. Resolve provider via `registry.resolve(capability_id)`
2. Call `provider.invoke(**kwargs)` directly
3. ToolExecutor emits tool-level events for audit

No persistent state. No lifecycle management.

---

### 7.3 Subagent Invocation (AGENT, CLI_AGENT)

1. Caller emits `capability.requested`
2. Kernel resolves deterministically
3. Kernel emits `capability.invoked`
4. Subagent receives task via adapter
5. Subagent may execute multi-step internal loop
6. Subagent emits `capability.completed`

Subagent invocation may involve session creation, task queuing, and lifecycle transitions.

---

## 8. Health & Liveness Model

### 8.1 Dual Health Model

**Stateless providers (TOOL, MCP_TOOL, SKILL):** `provider.health()` always returns `True`. No I/O. No background monitoring.

**Stateful providers (AGENT, CLI_AGENT):** Monitored by a background heartbeat task that polls `provider.health()` at a configurable interval. Cached health status is used by `resolve()` to skip unhealthy providers.

### 8.2 Health Transitions

When a stateful provider's health changes:

* `RegistryEvent(action="health_changed", ...)` is appended to the changelog
* Listeners are notified
* `resolve()` excludes unhealthy providers

### 8.3 Subagent Liveness

Subagent types (`AGENT`, `CLI_AGENT`) require mandatory liveness monitoring:

* Periodic heartbeat validation via `provider.health()`
* Automatic health status transitions in the registry
* Integration with lifecycle model (§10)

Tool types do not require persistent health monitoring.

## 10. Persistent Subagent Architecture

### 10.1 Motivation

Certain agent capabilities are provided by external CLI-based agents that:

* Maintain their own internal memory and session state
* Implement proprietary planning and execution strategies
* Provide interactive, incremental reasoning loops

Spawning them as short-lived subprocesses per request results in loss of session state, increased startup latency, and poor multi-step task continuity. Therefore, these agents are treated as **persistent CLI daemons** (`CLI_AGENT`).

### 10.2 Subagent Adapter Interface

A generic adapter layer manages external subagent processes:

```
SubagentAdapter
  ├── spawn()           → SubagentHandle
  ├── send(message)     → RequestID
  ├── receive()         → Response
  ├── health_check()    → HealthStatus
  ├── restart()         → SubagentHandle
  └── terminate()       → void
```

Responsibilities:

* Launch subagent process (in-process or daemon mode)
* Maintain persistent communication channel
* Serialize/deserialize structured messages
* Monitor liveness

### 10.3 Communication Protocol

Communication occurs over structured channels:

* **In-process subagents (AGENT):** Direct async method calls
* **External CLI subagents (CLI_AGENT):** stdin/stdout streams with JSON envelope

Envelope structure:

```json
{
  "request_id": "uuid",
  "task": "description",
  "payload": "...",
  "context": {...}
}
```

The adapter guarantees:

* Request-response correlation
* Timeout handling
* Partial streaming support (optional)

### 10.4 Session Multiplexing

Each subagent instance corresponds to one logical session.

The orchestrator maintains:

```
subagent_sessions: Dict[SessionID, SubagentHandle]
```

Session policies:

* One daemon per user session (default)
* Optional shared daemon pool (future extension)
* Explicit session termination required

### 10.5 Capability Registration for Subagents

`CLI_AGENT` capabilities must additionally declare:

* Supported task types
* `stateful: true` (derived from type)
* Concurrency model (serial, limited-parallel)
* Resource cost hints (memory, CPU class)

## 11. Subagent Lifecycle Model

### 11.1 States

```
CREATED → RUNNING → BUSY → IDLE → TERMINATED
```

State transitions:

* `CREATED → RUNNING`: Adapter calls `spawn()`
* `RUNNING → BUSY`: Task dispatched
* `BUSY → IDLE`: Task completed
* `IDLE → BUSY`: New task dispatched
* `IDLE → TERMINATED`: Explicit termination or timeout
* `BUSY → TERMINATED`: Forced termination (timeout/crash)

### 11.2 Failure Recovery

If a subagent becomes unresponsive:

1. Detect via `health_check()`
2. Attempt automatic restart via `restart()`
3. Optionally restore session context (best effort)
4. Escalate failure to orchestrator if restart fails

### 11.3 Scheduling

Subagents are treated as heavyweight workers:

* Avoid spawning per request; reuse active sessions
* Queue tasks if subagent is BUSY
* Enforce concurrency caps per user/session
* Apply resource limits (memory, CPU, time)

## 12. BaseAgent Capability Interface

BaseAgent gains a capability-aware interface:

```
declare_capabilities() -> list[CapabilityDescriptor]
```

Agents self-declare their capabilities. The orchestrator or registry can use this for introspection and discovery.

## 13. Version Negotiation

Semver rules apply:

* Major mismatch → incompatible
* Minor upgrade → compatible
* Patch upgrade → safe

Negotiation must be deterministic.

If no compatible version:

→ Emit `capability.unresolved`

## 14. Security Model

### 14.1 Access Control

Capabilities may declare:

```
required_roles
required_scopes
```

Kernel validates before invocation.

---

### 14.2 Sandboxing

Invocation isolation:

* Agent cannot mutate registry directly (only via register/unregister)
* Agent cannot bypass event bus for agent invocations
* No direct state mutation of other agents

### 14.3 Subagent Security

External CLI subagents introduce additional security surface:

* May call other tools internally
* May access filesystem
* May run long tasks

Security model for subagents (sandbox isolation, resource quotas, filesystem scoping, network restrictions) is defined in separate implementation RFCs. This RFC establishes the architectural boundary only.

## 15. Distributed Registry Model

For multi-node deployments:

Option A: Global event log
Option B: Partitioned registry with CRDT merge

If partitioned:

```
RegistryState = CRDTMerge(RegistryNodeStates)
```

Must converge deterministically.

## 16. Failure Modes

| Failure                | Resolution               |
| ---------------------- | ------------------------ |
| Agent crash            | Mark unavailable         |
| Duplicate registration | Idempotent via event ID  |
| Version conflict       | Deterministic resolution |
| Stale snapshot         | Replay / rebuild_from_events |
| Subagent crash         | Restart via adapter, escalate if repeated |
| Subagent timeout       | Force terminate, return error result |

## 17. Comparison: Capability Type Taxonomy

| Dimension | TOOL / MCP_TOOL / SKILL (Stateless) | AGENT / CLI_AGENT (Stateful) |
|-----------|------------------------------------|------------------------------|
| Nature | Function / procedure | Cognitive worker |
| Identity | Stateless capability | Stateful actor |
| Control | Caller retains full control | Delegated autonomy |
| State | None | Session-scoped, persistent |
| Temporal scope | Atomic, bounded | Long-lived, async |
| Failure impact | Single request | Session-level |
| Recovery | Retry call | Restart process |
| Scheduling | Spawn per call, no queuing | Reuse instance, task queue |
| Determinism | Deterministic (ideal) | May be nondeterministic |
| Health model | Always healthy | Heartbeat monitoring |
| Invocation | Direct `provider.invoke()` | Event-mediated |
| Security surface | Narrow, parameter validation | Broad, internal tool access |
| Observability | Log input/output | Reasoning trace, state snapshot, metrics |

**Cognitive boundary principle:**

* Tool call: "Do this operation for me."
* Subagent call: "Handle this problem domain for me."

The second implies delegation of **problem ownership**.

## 19. Comparison to Existing Systems

Traditional systems:

* Central registry (e.g., service mesh)
* Runtime RPC resolution
* Imperative routing

Noesium:

* **Provider-first registry** — live in-memory providers for fast reads
* **Hybrid event sourcing** — in-memory dict + async flush to EventStore; `rebuild_from_events()` for cold-start
* Projection-based discovery
* Deterministic resolution
* **Hybrid invocation** — direct for tools, event-mediated for agents
* Fully replayable routing decisions (via changelog)
* Persistent subagent lifecycle management
* Typed capability taxonomy (5 types: TOOL, MCP_TOOL, SKILL, AGENT, CLI_AGENT)

## 20. Guarantees

This RFC guarantees:

* Loose coupling between agents
* Deterministic capability resolution
* Replayable routing (via changelog)
* Version safety
* Health-aware invocation (dual model: always-true for stateless, heartbeat for stateful)
* Composable multi-agent scaling
* Persistent subagent session management
* Typed capability taxonomy (5 types)
* Provider-first design with events for audit and observability

## 22. Conclusion

RFC-1003 establishes the contract and discovery layer of Noesium.

It transforms:

Agent calls → Hybrid invocation (direct for tools, event-mediated for agents)
Service lookup → Provider-first registry with deterministic projection
Dynamic routing → Replayable resolution (changelog + rebuild_from_events)
External CLIs → Persistent subagent daemons (CLI_AGENT)

This enables a scalable, deterministic, auditable, and composable multi-agent ecosystem with first-class support for both stateless tools (TOOL, MCP_TOOL, SKILL) and stateful cognitive subagents (AGENT, CLI_AGENT).
