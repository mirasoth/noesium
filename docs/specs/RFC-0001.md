## 1. Abstract

This RFC defines the foundational architecture for a distributed, event-sourced, multi-agent system. The design targets long-running, stateful, autonomous agents capable of durable execution, delegation, replayability, and horizontal scaling.

The architecture is based on the following core decision:

> Each agent is implemented as an Event-Sourced Cognitive Kernel.

Rather than layering multiple orchestration frameworks, the system consolidates reasoning, delegation, and execution semantics into a single graph-based kernel per agent. Distribution is handled exclusively through an event bus. State is derived from immutable event logs.

This RFC formalizes the structural model, execution semantics, and scaling approach for this architecture.

**Implementation Status**: The current implementation uses LangGraph-based agents with mutable state and direct invocation patterns. The event-sourced architecture described in this RFC represents the foundational design principles. For the actual implementation, see RFC-1001 and RFC-1002. Event-sourced patterns are available as optional infrastructure for specific use cases (see Section 13: Alternative Architectural Patterns).

# 2. Architectural Goals

The system must support:

1. Long-running autonomous agents
2. Infinite or open-ended reasoning loops
3. Durable and resumable execution
4. Distributed multi-agent collaboration
5. Deterministic replayability
6. Horizontal scalability
7. Explicit separation of cognition, transport, and persistence

Non-goals:

* Direct RPC-based agent invocation
* Hidden subagent hierarchies
* Multiple overlapping orchestration layers

---

## 3. Core Principles

### 3.1 Single Execution Authority

Each agent contains exactly one execution authority: the Agent Kernel.

**Recommended Implementation**: LangGraph StateGraph provides graph-based execution with conditional branching, loops, checkpoints, and interrupt nodes. This serves as the single execution authority for all agent logic.

All reasoning, planning, branching, retries, tool invocation, and delegation logic occur inside this kernel.

No external orchestration layer is permitted to mutate agent state.

---

### 3.2 Mutable State with Event Optional

**Recommended Pattern**: Agents maintain state using mutable state objects managed by LangGraph's state mechanism.

Benefits:
- Simple mental model for developers
- Direct state access and mutation within graph nodes
- Efficient checkpoint-based persistence via LangGraph checkpointer
- Natural integration with Python ecosystem

**Alternative Pattern**: For use cases requiring strict audit trails and replay capabilities, agents MAY derive state from event projections (see Section 13: Alternative Architectural Patterns).

Formally:

State_t = Projection(Event_0 … Event_t)

This ensures:

* Replayability
* Deterministic reconstruction
* Auditable cognition
* Crash recovery

**Implementation Note**: The current implementation uses mutable state via LangGraph's TypedDict or Pydantic models. Event-sourced state is available as optional infrastructure when needed for audit or replay requirements.

---

### 3.3 Direct Delegation via Capability System

**Recommended Pattern**: Delegation occurs through direct method invocation via the SubagentManager and Capability Registry.

```python
# Direct delegation pattern
result = await subagent_manager.invoke(subagent_id, request, context)
```

Benefits:
- Clear call stack and error handling
- Synchronous or asynchronous execution models
- Type-safe request/response contracts
- Integrated with LangGraph workflow nodes

**Alternative Pattern**: For distributed multi-agent systems requiring loose coupling, delegation MAY be modeled as event emission to capability topics (see Section 13: Alternative Architectural Patterns).

**Implementation Note**: The current implementation uses direct invocation via SubagentManager. Event-based delegation is available as optional infrastructure for distributed scenarios.

---

### 3.4 Separation of Concerns

Cognition → Agent Kernel
Transport → Event Bus
Persistence → Event Store
Memory Projection → Projection Layer
Capability Routing → Capability Registry

No component duplicates responsibilities of another.

## 4. System Architecture

### 4.1 High-Level Topology

```
            Event Bus
                 |
---------------------------------
|                               |
```

Agent Kernel A                Agent Kernel B
|                               |
Event Store A                 Event Store B

Each agent node consists of:

1. Agent Kernel (graph-based runtime)
2. Event Store (optional, for event-sourced patterns)
3. Projection Engine (optional, for event-sourced patterns)
4. Capability Registry
5. Event Bus (for UI coordination)

**Implementation Note**: In the current implementation, the Event Bus (bubus) is used for UI/watchdog coordination, not for agent orchestration. Agents are invoked directly via method calls.

## 5. Component Specifications

### 5.1 Event Bus

Responsibilities:

* Topic-based routing
* Delivery guarantees (at-least-once minimum)
* Ordering within partitions
* Backpressure management

The bus is transport-only and contains no business logic.

Acceptable implementations:

* Kafka
* NATS
* Redis Streams

---

### 5.2 Agent Kernel

The Agent Kernel is a graph-based execution runtime implemented via LangGraph StateGraph.

**Core Capabilities**:
- **Directed Graph Execution**: Define agent workflows as state graphs with nodes and edges
- **Conditional Branching**: Route execution dynamically via `add_conditional_edges()`
- **Loop Constructs**: Implement iterative reasoning via graph cycles
- **Checkpointing**: Persist state at node boundaries via checkpointer (InMemorySaver, etc.)
- **Interrupt Nodes**: Pause execution for human-in-the-loop via `interrupt_before`
- **Async Execution**: Support both sync (`invoke()`) and async (`ainvoke()`) execution

**State Management**:
Agents maintain state using Pydantic models or TypedDict, updated through LangGraph's state mechanism. Each node receives the current state and returns a partial update.

```python
def my_node(state: AgentState) -> dict:
    # Process and return state delta
    return {"field": new_value}
```

**Tool and Subagent Integration**:
Tools and subagents are invoked directly within graph nodes. No event-mediated indirection is required.

```python
async def tool_node(state: AgentState) -> dict:
    result = await tool_executor.invoke(tool_name, arguments)
    return {"tool_result": result}
```

**Implementation Reference**: See RFC-1002 for detailed LangGraph agent implementation patterns.

---

### 5.3 Event Store

Each agent maintains an append-only event log containing:

* Inbound events
* Outbound events
* Internal transition events

Requirements:

* Immutable storage
* Ordered by logical time
* Indexed by event ID and correlation ID

---

### 5.4 Projection Layer

The projection layer derives current agent state from the event log.

Projection examples:

* Active task queue
* Subtask dependency graph
* Pending approvals
* Tool invocation status
* Working memory state

Projections must be deterministic.

## 6. Execution Semantics

### 6.1 Agent Execution Model

**Standard Pattern**: Agents are invoked directly and execute their LangGraph workflow.

```python
# Conversation agent
agent = AskuraAgent(config)
response = agent.start_conversation(user_id, message)

# Research agent
agent = TacitusAgent(config)
result = await agent.run(research_topic)
```

**Execution Flow**:
1. External caller invokes agent method (e.g., `run()`, `start_conversation()`)
2. Agent constructs initial state from inputs
3. LangGraph workflow executes via `graph.invoke(state, config)`
4. State flows through graph nodes
5. Final state is returned as result

**Checkpoint-Based Persistence**: LangGraph's checkpointer automatically persists state at node boundaries, enabling resume and recovery without explicit event emission.

**Event Processing Loop (Optional)**: For reactive agents that need to continuously process external events, a perpetual event loop MAY be implemented as a wrapper around the LangGraph workflow (see Section 13: Alternative Architectural Patterns).

---

### 6.2 Delegation Flow

**Standard Pattern**: Delegation uses SubagentManager with direct invocation.

```python
# Orchestrator agent delegates to subagent
manager = SubagentManager()
result = await manager.invoke(
    subagent_id="research_agent",
    request=SubagentInvocationRequest(
        message="Research quantum computing",
        context=SubagentContext(session_id="...", ...)
    )
)
```

**Delegation Steps**:
1. Orchestrator node calls SubagentManager with subagent ID and request
2. Manager looks up subagent provider from registry
3. Provider invokes subagent runtime (direct call, not event)
4. Subagent executes its LangGraph workflow
5. Result returned synchronously or via async stream

**Capability Resolution**: The SubagentManager uses the Capability Registry to resolve subagent providers. Topic-based subscription is not required.

**Event-Based Delegation (Optional)**: For distributed systems requiring loose coupling, delegation MAY be modeled as TaskRequested/TaskCompleted events (see Section 13: Alternative Architectural Patterns).

## 7. Multi-Agent Coordination Model

### 7.1 Capability-Based Routing

Agents declare capabilities by subscribing to topics.

Capabilities are logical, not physical.

Multiple agents may subscribe to the same topic, enabling:

* Load balancing
* Specialization
* Redundancy

---

### 7.2 Correlation and Causality

Each event must include:

* EventID
* CorrelationID
* CausationID
* Timestamp
* AgentID

This enables tracing and distributed workflow reconstruction.

---

## 8. Memory Architecture

Memory is divided into:

1. Working Memory (derived projection)
2. Persistent Semantic Memory (vector store / database)

All memory writes must be event-backed.

No silent state mutation is allowed.

---

## 9. Fault Tolerance

### 9.1 Crash Recovery

On restart:

1. Reload event log
2. Rebuild projections
3. Resume processing from last processed offset

No external coordination is required.

---

### 9.2 Idempotency

Event handlers must be idempotent.

Duplicate event delivery must not produce divergent state.

---

## 10. Scaling Model

Scaling is achieved via:

* Topic partitioning
* Horizontal agent replication
* Capability specialization
* Event partition ownership

Agents are independently deployable units.

## 13. Alternative Architectural Patterns

While the recommended approach uses mutable state and direct invocation, certain scenarios benefit from event-sourced patterns. This section documents optional alternative approaches.

### 13.1 Event-Sourced State Pattern

**When to Use**:
- Strict audit trail requirements
- Regulatory compliance (must replay history)
- Multi-agent collaboration with shared state
- Debugging complex state evolution

**How to Implement**:
1. Maintain EventStore per agent
2. Use ProjectionEngine to derive state from event log
3. Each state change emits a domain event
4. State_t = Projection(Event_0 ... Event_t)

**Infrastructure Available**:
- `EventEnvelope` (RFC-0002) - fully implemented
- `EventStore` (InMemoryEventStore, FileEventStore)
- `ProjectionEngine` with `BaseProjection`
- Domain event types in `noesium/core/event/types.py`

**Trade-offs**:
- ✅ Complete audit trail
- ✅ Deterministic replay
- ✅ Multi-agent state sharing
- ❌ Increased complexity
- ❌ Performance overhead
- ❌ Learning curve for developers

### 13.2 Perpetual Event Processing Loop

**When to Use**:
- Reactive agents responding to external stimuli
- Long-running autonomous agents
- Event-driven architectures

**How to Implement**:
```python
class ReactiveAgent:
    async def run_forever(self):
        while True:
            event = await self.event_bus.receive()
            await self.event_store.append(event)
            await self.projection_engine.apply_event(event)
            result = await self.kernel.execute(event)
            for emitted_event in result.events:
                await self.event_store.append(emitted_event)
                await self.event_bus.publish(emitted_event)
```

**Trade-offs**:
- ✅ Truly reactive behavior
- ✅ Event-driven coordination
- ❌ More complex than direct invocation
- ❌ Requires event bus setup

### 13.3 Event-Based Delegation

**When to Use**:
- Distributed multi-agent systems
- Loose coupling between agents
- Agents deployed across different processes/machines

**How to Implement**:
1. Orchestrator emits `TaskRequested` event to topic
2. Agents subscribe to capability topics
3. Agent processes task and emits `TaskCompleted`
4. Orchestrator receives completion event

**Event Types Available**:
- `TaskRequested` - request delegation
- `TaskCompleted` - delegation result
- `CapabilityInvoked` - capability invocation
- `CapabilityCompleted` - capability result

**Trade-offs**:
- ✅ Loose coupling
- ✅ Distributed coordination
- ✅ Load balancing across agents
- ❌ Asynchronous complexity
- ❌ Event ordering challenges
- ❌ Debugging difficulty

### 13.4 Choosing the Right Pattern

**Use Mutable State + Direct Invocation** (recommended) when:
- Building typical agent applications
- Developer simplicity is prioritized
- Agents run in single process
- Audit trail not required

**Use Event-Sourced State** when:
- Strict audit trail needed
- Regulatory compliance required
- Multi-agent state sharing needed

**Use Perpetual Event Loop** when:
- Agent must react to external events continuously
- Building reactive systems

**Use Event-Based Delegation** when:
- Agents deployed across multiple processes/machines
- Loose coupling is architectural requirement

## 14. Event Infrastructure Overview

While the recommended agent architecture uses direct invocation and mutable state, Noesium provides comprehensive event infrastructure for optional use cases.

### 14.1 Available Components

**EventEnvelope (RFC-0002)** - Fully Implemented
- Location: `noesium/core/event/envelope.py`
- 14-field canonical structure with UUIDv7, ISO8601 timestamps
- TraceContext for distributed tracing
- SignatureBlock for cryptographic verification
- Use when: Building event-sourced systems, audit trails, or distributed coordination

**EventStore** - Basic Implementation
- `InMemoryEventStore` - List-backed, for testing
- `FileEventStore` - JSONL file-backed, for persistence
- Use when: Persisting event log for replay or audit

**ProjectionEngine**
- Location: `noesium/core/projection/base.py`
- Derives state from event log via `fold()` operation
- Use when: Implementing event-sourced state pattern

**EnvelopeBridge**
- Location: `noesium/core/msgbus/bridge.py`
- Connects EventEnvelope to bubus EventBus
- Use when: Integrating event system with UI coordination

### 14.2 Integration Points

**Optional Usage**:
- EventEnvelope can wrap any state transition for audit
- EventStore can persist checkpoints alongside LangGraph checkpointer
- ProjectionEngine can derive read models from event log
- All components are available but not required for basic agent development

**bubus EventBus**:
- Used for UI/watchdog coordination, not agent orchestration
- Synchronous event dispatch for reactive UI updates
- Separate from agent execution flow
