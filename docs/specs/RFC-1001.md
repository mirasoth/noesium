# 1. Abstract

This RFC defines the event and progress streaming specification for the Noesium multi-agent system. The current implementation uses a lightweight approach with:

* **ProgressEvent** for TUI streaming
* **CognitiveContext** ([RFC-1010](RFC-1010.md)) for agent state continuity
* **Local changelog** in CapabilityRegistry for audit trails

The system uses the **bubus** event bus for in-process event dispatch.

---

# 2. Design Principles

## 2.1 Simplicity First

Events are lightweight dataclasses optimized for streaming and display.

## 2.2 Direct State Management

State is managed directly via LangGraph state and CognitiveContext rather than event-sourced projections.

## 2.3 Pragmatic Persistence

LangGraph checkpointer handles state persistence; no separate event store required.

---

# 3. ProgressEvent

The primary event type for TUI streaming:

```python
@dataclass
class ProgressEvent:
    type: ProgressEventType
    message: str
    metadata: dict[str, Any] = field(default_factory=dict)
    timestamp: datetime = field(default_factory=lambda: datetime.now(tz=timezone.utc))
```

### 3.1 Event Types

```python
class ProgressEventType(str, Enum):
    THINKING = "thinking"
    TOOL_CALL = "tool_call"
    TOOL_RESULT = "tool_result"
    SUBAGENT_START = "subagent_start"
    SUBAGENT_PROGRESS = "subagent_progress"
    SUBAGENT_END = "subagent_end"
    TEXT = "text"
    ERROR = "error"
    COMPLETE = "complete"
```

### 3.2 Usage

```python
# Emit progress during agent execution
await emit_progress(ProgressEvent(
    type=ProgressEventType.THINKING,
    message="Analyzing the request...",
))

# Tool call progress
await emit_progress(ProgressEvent(
    type=ProgressEventType.TOOL_CALL,
    message="Calling web_search",
    metadata={"tool": "web_search", "args": {"query": "..."}},
))
```

---

# 4. Event Bus Integration

The system uses **bubus** for event dispatch:

```python
from bubus import EventBus

bus = EventBus()
bus.subscribe(ProgressEvent, handler)
await bus.publish(event)
```

Events are dispatched asynchronously and handlers are invoked in subscription order.

---

# 5. Capability Registry Changelog

The CapabilityRegistry maintains a local append-only changelog for audit:

```python
@dataclass
class RegistryEvent:
    action: str  # "registered" | "unregistered" | "health_changed"
    capability_id: str
    version: str
    capability_type: str
    timestamp: float
    detail: dict[str, Any]
```

This provides audit trail without external EventStore dependency.

---

# 6. Agent State Continuity

Agent state continuity is handled by:

1. **CognitiveContext** ([RFC-1010](RFC-1010.md)) - Cross-turn cognitive state
2. **LangGraph Checkpointer** - Graph state persistence

See [RFC-1010](RFC-1010.md) for CognitiveContext specification.

---

# 7. Tracing and Observability

Tracing is handled via:

* **LangGraph callbacks** - Node execution callbacks
* **OpenTelemetry** - Distributed tracing when needed
* **Opik integration** - Token tracking and cost monitoring

No custom TraceContext structure required.

---

# 8. Future Considerations

Full event-sourced architecture with EventEnvelope may be future work if:

* Strict audit requirements emerge
* Distributed multi-agent coordination needed
* Regulatory compliance requires complete audit trails
* Deterministic replay from event log required

Current implementation prioritizes simplicity and developer ergonomics.

---

# 9. Summary

The Noesium event system uses:

| Component | Purpose |
|-----------|---------|
| ProgressEvent | TUI streaming |
| bubus EventBus | In-process dispatch |
| RegistryEvent | Capability audit |
| CognitiveContext | State continuity |
| LangGraph Checkpointer | Persistence |

This lightweight approach provides observability without the complexity of full event sourcing.
