## Objective

Fix the error handling when LLM providers reject content due to policy violations (e.g., Dashscope's `data_inspection_failed` error). The issue was that the system was retrying requests that would always fail due to content policy violations, wasting time and providing poor user feedback.

## Problem

When using Dashscope/Alibaba Cloud as the LLM provider, queries about sensitive topics (e.g., "US-Iran war") would fail with `data_inspection_failed` errors. The error was:

```
Error code: 400 - {'error': {'message': '<400> InternalError.Algo.DataInspectionFailed: Input text data may contain inappropriate content.'}}
```

The system would retry this error multiple times (default 2-3 attempts), which was futile since content filtering errors are not transient.

## Notes

### Error Detection Patterns

The following error patterns are now detected as non-retryable content filter errors:
- `data_inspection_failed` - Dashscope/Alibaba Cloud
- `inappropriate content` - Dashscope/Alibaba Cloud
- `content_filter` - OpenAI and compatible APIs
- `content_policy` - Generic content policy errors
- `safety` + `violation` - Safety violation errors

### User-Friendly Error Message

When a content filter error occurs, users now see:

```
⚠️ Content Policy Violation

Your request was blocked by [provider]'s content safety system. This can happen with sensitive topics like wars, conflicts, or other content that may violate usage policies.

Suggestions:
• Rephrase your query in more neutral terms
• Try using a different LLM provider (e.g., switch from Dashscope to OpenAI)
• Use /research or /deep_research commands for web-based information gathering

Technical details: [error message]
```
