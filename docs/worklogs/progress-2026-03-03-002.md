## Objective

Complete implementation of the provider-first, hybrid event-sourced capability registry design (RFC-0005 / RFC-1004) across all layers: specifications, core capability module, base agent interface, NoeAgent integration, and test suites.

## Completed

### Specifications
- Revised RFC-0005 to reflect provider-first hybrid design: 5-type taxonomy, CapabilityProvider protocol, unified CapabilityRegistry, dual health model, hybrid invocation
- Created RFC-1004 (Capability Registry Implementation Architecture) with full module structure, core types, provider adapters, registry API, invocation flow, health model, and testing strategy
- Updated rfc-history.md and rfc-index.md with new entries

### Core Capability Module (`noesium/core/capability/`)
- Updated `__init__.py` to export new types (CapabilityType, CapabilityDescriptor, CapabilityProvider, CapabilityQuery, all provider adapters, CapabilityRegistry, RegistryEvent)
- Deleted obsolete `discovery.py` (DiscoveryService) and `resolution.py` (DeterministicResolver)

### Event Types (`noesium/core/event/types.py`)
- Added `CapabilityUnregistered`, `CapabilityRequested`, `HealthChanged` event types
- Extended `CapabilityRegistered`, `CapabilityInvoked`, `CapabilityCompleted` with `capability_type`, `correlation_id`, `duration_ms` fields

### Base Agent Interface (`noesium/core/agent/base.py`)
- Added `declare_capabilities() -> list[CapabilityDescriptor]` to `BaseAgent` (default empty list)
- Updated docstrings to reference RFC-0005 §12

### NoeAgent Integration
- Rewrote `noesium/noe/agent.py` to use `CapabilityRegistry` as single source of truth
- Rewrote `noesium/noe/nodes.py`:
  - `_build_tool_descriptions()` reads from registry providers instead of ToolRegistry
  - `execute_step_node()` accepts `registry` parameter instead of `tool_registry`
  - `tool_node()` invokes providers via `registry.get_by_name()` and `provider.invoke()`
  - Type coercion via `_coerce_tool_args()` against descriptor schemas

### Tests
- Rewrote `tests/capability/test_models.py` (12 tests for enums, descriptor, query)
- Created `tests/capability/test_providers.py` (17 tests for all 5 provider adapters)
- Rewrote `tests/capability/test_registry.py` (20 tests: registration, discovery, resolution, health, listeners, hybrid ES)
- Deleted obsolete `tests/capability/test_resolution.py`
- Rewrote `tests/integration/test_capability_flow.py` (7 tests: register-discover-resolve, unregister, mixed types, health-aware, audit trail, rebuild, listener)
- Rewrote `tests/agents/test_noe_agent.py` to use CapabilityRegistry mocks (31 non-LLM tests)

## Notes

- Health cache is keyed by `capability_id`, not per-provider-instance. Multiple versions of the same capability share a single health status. This is intentional for simplicity but could be refined if per-version health becomes necessary.
- Old `DiscoveryService` and `DeterministicResolver` are fully removed; all references in code imports are gone. Documentation references in spec files are kept for historical context.
- The `ToolRegistry` in `noesium/core/toolify/tool_registry.py` still exists as a lower-level construct; NoeAgent no longer uses it directly — everything goes through `CapabilityRegistry`.
