## Objective
Fix the issue where the `/browser` command in NoeAgent TUI hangs at "Starting..." without showing any progress updates, and investigate why TacitusAgent was being invoked instead of BrowserUseAgent.

## Completed

1. **Diagnosed the root cause**: The `_process_subagent_command` function was not consuming streaming progress events. It displayed "Starting..." and then blocked until `execute_subagent_command` returned, without showing any intermediate progress.

2. **Fixed `_process_subagent_command` in `noesium/noeagent/tui.py`**:
   - Replaced blocking `execute_subagent_command` call with direct streaming from `provider.invoke_streaming`
   - Added `Live` display to show real-time progress updates
   - Added proper handling for `SUBAGENT_PROGRESS`, `SUBAGENT_END`, and `ERROR` events
   - Added error handling with user-friendly error messages
   - Removed unused `execute_subagent_command` import

3. **Improved `BrowserUseAgent.astream_progress` message**:
   - Changed "Starting browser session..." to "Launching browser... (this may take up to 30s)" to better inform users about expected wait time

4. **Fixed config loading bug in `noesium/noeagent/config.py`**:
   - The `from_global_config` method was dropping critical fields when creating `AgentSubagentConfig` objects
   - Added: `requires_explicit_command`, `task_types`, `use_cases`, `keywords`, `preferred_for`
   - This was causing `tacitus` to have `requires_explicit_command=False` instead of `True`, allowing the LLM to auto-route to tacitus when it shouldn't

## Notes

**Root Cause Analysis for "Starting..." hang**:
The original `_process_subagent_command` function had infrastructure for displaying progress (spinner, activity_lines), but never used it. It called `execute_subagent_command` which blocked until completion, queuing events to `_subagent_event_queue` that were never consumed.

**Root Cause Analysis for TacitusAgent being invoked**:
The `from_global_config` method in `config.py` was NOT passing `requires_explicit_command` through when creating `AgentSubagentConfig` objects. This caused:
1. `tacitus` to have `requires_explicit_command=False` (default) instead of `True`
2. The LLM could auto-route queries to `tacitus` even without explicit `/research` command
3. When the browser_use agent had issues (slow/failed startup), the system might have been falling back to tacitus via LLM routing

**Browser Startup Behavior**:
- Browser launch can take up to 30 seconds (hardcoded timeout in `_wait_for_cdp_url`)
- The underlying Agent class doesn't support streaming progress during browser startup
- Progress events are only yielded after the browser is fully started

**Why TacitusAgent appeared in logs**:
Looking at the user's logs, the timing suggests that when the browser was taking a long time to start (or failing silently), the LLM might have been routing to tacitus as a fallback for web research tasks. The missing `requires_explicit_command=True` for tacitus allowed this auto-routing.
