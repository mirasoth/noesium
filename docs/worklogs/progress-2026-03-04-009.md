## Objective

Analyze why Tavily appears not to work when using the web_search toolkit with config override, and ensure config (including API key) is applied correctly.

## Analysis Summary

### 1. Config override **is** working

From your log:

- `Search engine registry initialized with 9 engines: ['baidu', 'brave', 'duckduckgo', 'googleai', 'searxng', 'tavily', 'wechat', 'bing', 'google']` — this is the **wizsearch library** discovering all engines; it does not reflect our config.
- `Starting WizSearch for query: '...' with engines: ['tavily', 'duckduckgo']` — this is the **actual** engine list used for the search and comes from our `WizSearchConfig(enabled_engines=...)`.

So the engine list we pass from the toolkit (`enabled_engines` from `ToolkitConfig.config`) is correctly applied: WizSearch uses only `['tavily', 'duckduckgo']`. If you want **only** Tavily, set in your config:

```json
"toolkit_configs": {
  "wizsearch": {
    "enabled_engines": ["tavily"]
  }
}
```

If you see `['tavily', 'duckduckgo']`, that is because the config that was loaded has both (e.g. from a different config file or test fixture that uses `["tavily", "duckduckgo"]`).

### 2. Why Tavily can “not work” (no results / silent failure)

- The **wizsearch** package (PyPI `wizsearch>=1.0.1`) does **not** accept API keys in `WizSearchConfig`. It only has: `enabled_engines`, `max_results_per_engine`, `timeout`, `fail_silently`.
- When WizSearch creates engines, it calls `SearchEngineRegistry.create_engine(engine_name, max_results=...)` with **no** API key or other auth. So engine-specific config (e.g. Tavily API key) cannot be passed through WizSearch.
- For **Tavily**, wizsearch uses `LangchainTavilySearch` (from `langchain_tavily`). That integration reads **only** the `TAVILY_API_KEY` **environment variable**. There is no API key parameter in `TavilySearchConfig` in the wizsearch source (`.venv/.../wizsearch/engines/tavily_search.py`).
- With `fail_silently=True` (our default), if Tavily fails (e.g. missing or invalid `TAVILY_API_KEY`), WizSearch catches the exception, logs it, and continues with the other engines (e.g. DuckDuckGo). So you can see “Performing Tavily search for query: …” in the log but get **no** Tavily results and only DuckDuckGo results — Tavily is “not working” even though it is in the engine list.

So: **Tavily “not working” is almost certainly due to `TAVILY_API_KEY` not being set (or not set correctly) in the environment when the wizsearch library runs**, and our toolkit did not previously forward an API key from config to the environment.

### 3. Third-party wizsearch source (reference)

There is no `thirdparty/wizsearch` in this repo; the code comes from the installed package, e.g.:

- `.venv/lib/python3.11/site-packages/wizsearch/wizsearch.py`  
  - `WizSearchConfig`: only `enabled_engines`, `max_results_per_engine`, `timeout`, `fail_silently`.  
  - `SearchEngineRegistry.create_engine(name, max_results, **kwargs)` — we only pass `max_results` from our side; no API key.  
  - If `enabled_engines is None`, the library defaults to all 9 engines; we always pass a list, so our override is used.
- `.venv/lib/python3.11/site-packages/wizsearch/engines/tavily_search.py`  
  - Uses `LangchainTavilySearch` with no API key argument; Tavily relies on `TAVILY_API_KEY` in the environment.

## Completed

- Confirmed config override: `enabled_engines` from toolkit config is passed to `WizSearchConfig` and WizSearch uses that list.
- Identified root cause of Tavily not working: API key not available to the library; env var not set from config.
- Implemented: forward `TAVILY_API_KEY` (and optional `api_key`) from toolkit config to `os.environ` before creating WizSearch or TavilySearch, so config file / NoeConfig can supply the key without requiring a separate env export.
