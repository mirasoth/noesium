## Objective

Per RFC-1006 design, subagent browser_use progress details were not visible on the TUI. Analyze root causes and fix.

## Completed

- Diagnosed three bugs blocking browser_use progress details from appearing in TUI
- Fixed `NoeBuiltinSubagentRuntime.execute()` to translate all `ProgressEventType` variants from `browser_use.astream_progress` into richly-typed `SubagentProgressEvent`s with full metadata
- Fixed `_subagent_event_to_progress()` to preserve `metadata`, `tool_name`, `tool_args`, `tool_result`, `step_index`, `step_desc`, `plan_snapshot` when converting `SubagentProgressEvent` → `ProgressEvent`
- Added `SUBAGENT_TOOL_CALL` → `TOOL_START` and `SUBAGENT_TOOL_RESULT` → `TOOL_END` mappings in `_subagent_event_to_progress()` (previously these event types were silently dropped)

## Root Cause Summary

**Bug 1 — `NoeBuiltinSubagentRuntime.execute()` stripped all rich event types:**
`browser_use.astream_progress` emits `PLAN_CREATED`, `STEP_START`, `STEP_COMPLETE`, `TOOL_START`, `TOOL_END`, and `THINKING` events. The old code only detected `FINAL_ANSWER` and collapsed everything else into a bare `SubagentProgressEvent.progress()` with no metadata. TUI's `SubagentTracker` and detail rendering rely on `child_event_type`, `agent_type`, `step_index`, `plan_snapshot` to display meaningful progress.

**Bug 2 — `_subagent_event_to_progress()` dropped all structured fields:**
The conversion from `SubagentProgressEvent` → `ProgressEvent` only copied `summary` and `detail`, discarding `metadata`, `tool_name`, `step_index`, `plan_snapshot`. The TUI's `SUBAGENT_PROGRESS` handler at `tui.py:924` reads `event.metadata["child_event_type"]` and `event.plan_snapshot`, both of which were always `None`/empty.

**Bug 3 — Missing `SUBAGENT_TOOL_CALL`/`SUBAGENT_TOOL_RESULT` in type_map:**
`SubagentEventType.SUBAGENT_TOOL_CALL` and `SUBAGENT_TOOL_RESULT` had no mapping to `ProgressEventType`, causing them to return `None` and be silently discarded rather than forwarded as `TOOL_START`/`TOOL_END` events.

## Files Changed

- `noesium/noeagent/subagent_runtimes.py` — Rewrote `NoeBuiltinSubagentRuntime.execute()` astream_progress branch to emit typed `SubagentProgressEvent`s (tool_call, tool_result, progress with plan/step/thinking metadata) for all `ProgressEventType` variants from browser_use
- `noesium/noeagent/agent.py` — Fixed `_subagent_event_to_progress()` to: (a) add TOOL_CALL/TOOL_RESULT mappings, (b) extract and forward payload fields (child_event_type, agent_type, step_index, step_desc, plan_snapshot) into the resulting ProgressEvent's metadata/fields

## Notes

- The `payload` dict on `SubagentProgressEvent` is used as the carrier for structured metadata between the runtime adapter and `_subagent_event_to_progress`. Fields `child_event_type`, `agent_type`, `step_index`, `step_desc`, `plan_snapshot` are placed in `payload` by the runtime, then unpacked in the conversion function.
- `SUBAGENT_WARNING`, `SUBAGENT_HITL_REQUEST`, `SUBAGENT_HITL_RESPONSE` remain unmapped (they are not supported on the non-HITL browser_use path and have no `ProgressEventType` equivalent yet).
